总结：

 当用户在浏览器输入地址（URL）并回车后，首先浏览器会解析 URL 得到协议、域名等信息，先查本地 DNS 缓存/Hosts 文件，未命中则向本地 DNS 服务器发起递归查询，最终从权威 DNS 拿到目标服务器 IP；同时判断是否为 HTTPS，若是则先进行 TCP 三次握手建立连接，再走 TLS/SSL 握手协商加密密钥。之后浏览器按 HTTP/HTTPS 协议组装请求报文发送给服务器，服务器处理（可能查数据库、调接口等）返回 HTML 响应，浏览器开始解析 HTML 构建 DOM 树和 CSSOM 树、合成渲染树，执行 JS 脚本，进行布局（Reflow）与绘制（Paint），最终将像素点显示在屏幕上，若有异步请求还会继续拉取数据更新页面。

## 1. 用户输入 URL

比如用户输入：

```
https://www.example.com/index.html
```

浏览器会先解析这个 URL，得到几个关键信息：

```
协议：https域名：www.example.com端口：443路径：/index.html
```

如果用户只输入：

```
www.example.com
```

浏览器可能会自动补全成：

```
https://www.example.com
```

如果输入的是普通文字，浏览器可能会直接走搜索引擎。

## 2. 检查缓存

浏览器不会一上来就直接请求服务器，而是会先检查缓存。

主要包括：

```
浏览器缓存
DNS缓存（缓存域名对应的ip）
操作系统缓存
本地hosts文件
路由器或运营商DNS缓存
```

如果资源命中**强缓存**（浏览器自己判断缓存还没过期，直接用本地缓存，不请求服务器；与之对应的就是协商缓存），浏览器可能直接使用本地缓存，不再请求服务器。

如果缓存过期，浏览器会向服务器发起协商缓存请求，比如通过：

```
If-Modified-SinceIf-None-Match
```

服务器如果判断资源没有变化，会返回：

```
304 Not Modified
```

浏览器继续使用本地缓存。

## 3. DNS 解析

如果需要访问网络，浏览器需要知道服务器 IP 地址。

因为浏览器不能直接通过域名访问服务器，需要先把：

```
www.example.com
```

解析成类似：

```
93.184.216.34
```

DNS 查询大致顺序是：

```
浏览器DNS缓存
操作系统DNS缓存
hosts文件
本地DNS服务器
根DNS服务器
顶级域名服务器
权威DNS服务器
```

最终拿到目标服务器的 IP 地址。

## 4. 建立 TCP 连接

拿到 IP 后，浏览器会和服务器建立 TCP 连接。

TCP 建立连接需要三次握手：

```
第一次：客户端发送 SYN
第二次：服务器返回 SYN + ACK
第三次：客户端发送 ACK
```

三次握手完成后，客户端和服务器之间就建立了一条可靠的 TCP 连接。

## 5. HTTPS 还需要 TLS 握手

如果访问的是 HTTPS，在 TCP 连接之后，还会进行 TLS 握手。

TLS 握手主要做几件事：

```
确认加密协议版本
协商加密算法
服务器发送数字证书
浏览器验证证书是否合法
双方生成会话密钥
```

完成 TLS 握手后，后续 HTTP 数据就会被加密传输。

所以 HTTPS 可以简单理解为：

```
HTTP + TCP + TLS
```

## 6. 浏览器发送 HTTP 请求

连接建立完成后，浏览器会发送 HTTP 请求。

请求内容大致包括：

```
GET /index.html HTTP/1.1Host: www.example.comUser-Agent: ChromeCookie: xxxAccept: text/html
```

常见的请求方法有：

```
GET：获取资源POST：提交数据PUT：更新资源DELETE：删除资源
```

## 7. 服务器处理请求

服务器收到请求后，一般会经过这些步骤：

```
负载均衡
Web 服务器应用服务器
业务逻辑处理
数据库查询
缓存查询
生成响应结果
```

比如一个 Java 后端系统，可能会经过：

```
Nginx → Tomcat / Spring Boot → Controller → Service → DAO → MySQL / Redis
```

最后服务器把处理结果封装成 HTTP 响应返回给浏览器。

## 8. 服务器返回 HTTP 响应

HTTP 响应通常包括三部分：

```
状态行
响应头
响应体
```

例如：

```
HTTP/1.1 200 OKContent-Type: text/htmlContent-Length: 1024<html>...</html>
```

常见状态码：

```
200：请求成功301 / 302：重定向304：资源未修改，使用缓存400：客户端请求错误401 / 403：未认证或无权限404：资源不存在500：服务器内部错误502 / 503：网关或服务不可用
```



浏览器拿到 HTML 后，并不是直接显示页面，而是要经过渲染流程。

## 9. 解析 HTML，生成 DOM 树

浏览器会把 HTML 解析成 DOM 树。

例如：

```
<html>
  <body>
    <div>Hello</div>
  </body>
</html>
```

会被解析成类似：

```
Document
 └── html
     └── body
         └── div
```

DOM 树描述的是页面结构。

## 10. 解析 CSS，生成 CSSOM 树

浏览器还会解析 CSS，生成 CSSOM 树。

CSSOM 描述的是元素的样式，比如：

```
div{
  color: red;
  font-size: 16px;
}
```

## 11. 生成渲染树 Render Tree

浏览器会把 DOM 树和 CSSOM 树结合起来，生成渲染树。

渲染树只包含需要显示的节点。

比如：

```
display: none;
```

对应的元素不会出现在渲染树中。

## 12. 布局 Layout

布局阶段也叫 reflow。

浏览器会计算每个元素在页面中的位置和大小，比如：

```
宽度是多少
高度是多少
在页面的哪个位置
和其他元素如何排列
```

## 13. 绘制 Paint

绘制阶段，浏览器会把元素的颜色、文字、边框、阴影、背景等画出来。

## 14. 合成 Composite

现代浏览器会把页面拆成多个图层。

最后通过合成线程把不同图层合成在一起，并显示到屏幕上。





### 强缓存与协商缓存

强缓存的意思是浏览器自己判断缓存还没过期，直接用本地缓存，不请求服务器；而协商缓存是浏览器本地有缓存，但是不确定还能不能用，所以要问服务器。

**强缓存**常见响应头有两个：Cache-Control    Expires

**Cache-Control**

比如服务器返回：Cache-Control: max-age=31536000    意思是这个资源可以缓存这么多秒，在这个时间内，浏览器再次请求这个资源时，可以直接使用本地缓存，不需要再访问服务器

Cache-Control常见的参数有：

max-age=3600（同上）：表示资源缓存这么多秒

no-cache：这个名字容易误解，它不是不缓存，而是说可以缓存，但每次使用前都必须问服务器有没有更新，也就是它会走**协商缓存**

no-store：这个才是真正的不缓存

public：表示资源可以被浏览器、CDN、代理服务器等缓存

private：表示资源只能被用户自己的浏览器缓存，不能被 CDN 或代理服务器共享缓存。

**协商缓存**主要靠两组字段。

第一组：Last-Modified / If-Modified-Since

服务器第一次返回资源时会带上：

```http
Last-Modified: Tue, 23 Jun 2026 10:00:00 GMT
```

表示这个资源最后修改时间，浏览器下次请求时，会带上：

```http
If-Modified-Since: Tue, 23 Jun 2026 10:00:00 GMT
```

意思是我本地这个资源最后修改时间是这个，你看看你服务器上的资源有没有更新。如果没有更新，服务器返回304 Not Modified；如果更新了，服务器返回200 ok和新的资源。

第二组：ETag / If-None-Match

服务器第一次返回资源时，会带上：

```http
ETag: "abc123"
```

可以理解为资源的唯一标识，类似文件指纹，浏览器下次请求时会带上：

```http
If-None-Match: "abc123"
```

服务器会判断这个ETag是否还匹配。如果匹配，说明资源没变：304 Not Modified；如果不匹配，说明资源变了：200 OK 并返回新资源

#### 追问：ETag 和 Last-Modified 的区别

Last-Modified 依赖文件修改时间，有几个缺点：

```
1. 时间精度通常是秒级，如果一秒内修改多次，可能识别不到
2. 文件内容没变，但修改时间变了，也会导致缓存失效
3. 分布式服务器之间时间可能不一致
```

ETag 是根据资源内容生成的标识，更精确。

所以一般来说：

```
ETag 优先级高于 Last-Modified
```

面试可以这样说：

> Last-Modified 是根据资源最后修改时间判断缓存是否可用，而 ETag 是根据资源内容生成唯一标识。ETag 精度更高，可以解决 Last-Modified 秒级精度不足、文件时间变化但内容未变等问题。因此在两者同时存在时，服务器通常优先使用 ETag。



#### 追问：强缓存和协商缓存的整体流程

```textile
1. 浏览器请求一个资源
2. 检查 Cache-Control / Expires，判断强缓存是否有效
3. 如果强缓存有效，直接从内存缓存或磁盘缓存读取
4. 如果强缓存失效，浏览器带上 If-None-Match 或 If-Modified-Since 请求服务器
5. 服务器判断资源是否变化
6. 没变化返回 304，浏览器继续使用本地缓存
7. 有变化返回 200，并返回新的资源内容
```
