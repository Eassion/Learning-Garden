在使用消息队列（MQ）时，**消息重复投递几乎无法完全避免**。

无论是 Broker 重试、消费者重启，还是网络抖动，都可能导致同一条消息被消费多次。因此，大多数消息队列都只能保证 **At Least Once（至少消费一次）**，而不是 **Exactly Once（恰好消费一次）**。

因此，**幂等通常需要在消费端实现。**

## 什么是幂等？

幂等（Idempotent）指的是：

> **同一条消息处理一次和处理多次，最终结果完全一致。**

例如：

- 重复创建订单 ❌
- 重复扣库存 ❌
- 重复扣余额 ❌

这些业务如果没有幂等控制，就会产生严重问题。

# 常见实现方式

## 1. 消息唯一 ID 去重（最常用）

每条消息都携带一个全局唯一的 Message ID。

消费者收到消息后：

1. 查询该 Message ID 是否已经处理过；
2. 已处理，则直接返回；
3. 未处理，则执行业务逻辑；
4. 业务成功后，将 Message ID 保存起来。

流程如下：

```text
收到消息
    │
    ▼
查询 MessageID 是否存在
    │
 ┌──┴──┐
 │     │
存在   不存在
 │      │
跳过   执行业务
 │      │
 │   保存 MessageID
 └──────┘
```

这是实际开发中最通用的方案。

---

## 2. 数据库唯一索引

如果业务本身存在天然唯一键，可以直接利用数据库约束实现幂等。

例如：

- 订单号（orderId）
- 支付流水号
- 用户 + 活动 ID

例如：

```sql
CREATE UNIQUE INDEX idx_order
ON order(order_no);
```

第一次插入成功。

第二次插入：

```
Duplicate Key
```

数据库直接拦截重复数据。

### 优点

- 实现简单
- 非常可靠
- 数据一致性最好

### 缺点

- 每次都需要访问数据库
- 并发高时压力较大

---

## 3. 状态机控制

很多业务本身就是一个状态流转过程。

例如订单：

```
待支付
    │
    ▼
已支付
    │
    ▼
已发货
    │
    ▼
已完成
```

假设收到一条"支付成功"消息：

正常情况：

```
待支付
    ↓
已支付
```

如果再次收到同一条消息：

```
已支付
```

状态没有变化，因此直接忽略。

这种方式特别适用于：

- 订单
- 支付
- 审批
- 工单

等具有明确生命周期的业务。

---

## 4. Redis 去重

利用 Redis 的高性能，可以快速判断消息是否已经消费。

常见方式：

```text
SETNX messageId
```

如果返回：

```
true
```

说明第一次消费，可以继续执行。

如果返回：

```
false
```

说明已经消费过，直接跳过。

### 优点

- 速度快
- 并发能力强
- 非常适合高吞吐 MQ

### 缺点

Redis 只能作为缓存层。

例如：

```
Redis 写成功
↓

数据库还没写成功

↓

程序宕机
```

下一次消息来了：

Redis 已存在记录

数据库实际上却没有完成业务。

因此：

> **Redis 去重必须配合数据库作为最终兜底。**

---

## 5. 乐观锁（Version）

对于更新操作，可以增加版本号（Version）。

例如：

```text
version = 1
```

更新时：

```sql
UPDATE product
SET stock = stock - 1,
    version = version + 1
WHERE id = 1
AND version = 1;
```

如果返回：

```
更新 1 行
```

说明成功。

如果返回：

```
更新 0 行
```

说明数据已经被别人修改，本次更新直接放弃。

这种方式广泛应用于：

- 库存扣减
- 秒杀
- 高并发更新

# 各方案对比

| 方案            | 实现难度 | 性能   | 推荐场景          |
| ------------- | ---- | ---- | ------------- |
| Message ID 去重 | ⭐⭐   | ⭐⭐⭐  | 通用 MQ 消费      |
| 数据库唯一索引       | ⭐    | ⭐⭐   | 新增类业务（订单、支付等） |
| 状态机           | ⭐⭐   | ⭐⭐⭐  | 有状态流转的业务      |
| Redis 去重      | ⭐⭐   | ⭐⭐⭐⭐ | 高并发场景         |
| 乐观锁 Version   | ⭐⭐⭐  | ⭐⭐⭐  | 更新类业务、库存扣减    |

---

# 实际项目中的建议

实际生产环境中，通常不会只采用一种方案，而是组合使用。

例如：

- **订单创建**：数据库唯一索引
- **库存扣减**：乐观锁 + 数据库事务
- **MQ 消费**：Message ID 去重 + Redis 缓存
- **订单状态更新**：状态机控制

对于**金额、库存等敏感操作**，不要只依赖 Redis，因为 Redis 无法保证最终一致性。

> **一定要有数据库层面的兜底（唯一索引、事务、乐观锁等），确保数据安全。**
