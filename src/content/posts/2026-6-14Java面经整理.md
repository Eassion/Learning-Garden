### 一、Java基础

#### 1.==和equals()的区别

省流：==比较的是“物理层面“（内存地址或数值），而equals()通常用来比较”逻辑层面（对象的内容）“

**（1）==运算符**

它的行为取决于它两边变量的数据类型：

- 基本数据类型

如int，double，boolean，char等，比较的是它们实际的数值是否相等

- 引用数据类型

如String，数组，对象等，比较的是对象的内存地址是否相同，只有在两个引用指向堆内存中的同一个对象实例时，才会返回true

**（2）equals()**

它是java.lang.Object类中的一个方法。因为所有Java类都继承自Object，所以所有对象都有这个方法

- 默认行为

在Object类的源码中，equals()方法的内部实现其实就是return（this==obj）。也就是说其实内部还是使用==，默认比较的也是内存地址

- 自定义类重写equals()

Java中大多数内置的核心类（String，Integer，Date等）都重写了这个方法。重写后，它的作用变成了比较对象内部包含的数据是否相等，而不再关心它们是否是内存中的同一个对象。对于你自定义的类，如果想要比较内容，也需要手动重写，通常配合重写hashCode()。

### 二、String、StringBuilder、StringBuffer的区别

瞄准三个点：可变性、线程安全和效率

- String

不可变，一旦创建（通常在字符串常量池或堆内存）中被创建，它的值就不可变了。

底层实现：Java8及之前：final char[]；Java9之后：final byte[]+coder

性能：每次对String进行拼接、裁剪操作时，由于不可变，它就在堆内存中创建一个全新的String对象，如果在循环中频繁地进行这种操作，会产生大量废弃的临时对象，给GC带来巨大压力，严重拖慢性能。

线程安全：由于不可变，每次操作都会指向不同的对象，所以天然线程安全

- StringBuilder

可变。

底层实现：它底层同样维护着一个字符数组，但是没有final修饰，当你向它追加内容时（通过append()方法），它是直接在原有的数组上进行修改。如果容量不够了，它会自动扩容，通常是当前容量的2倍+2，并将旧数据拷贝到新数组中。

性能：由于每次操作不需要重新创建对象，在频繁地操作字符串时，性能极高

线程不安全：它的内部方法没有没有加任何同步锁，在多线程环境下操作同一个StringBuilder可能会导致数据错乱

- StringBuffer

可变。

底层实现：你可以把它看成是StringBuilder的”线程安全版本“，底层结构和扩容机制几乎一样。唯一的区别在于，它的核心方法（append()、insert()等）都加上了synchronized关键字进行同步

性能：因为加了同步锁，多线程访问时需要排队，所以效率比StringBuilder要低，但比频繁创建新对象的String还是要快得多

线程安全：绝对的线程安全，适合多线程环境下共享并修改同一个字符串

###### 追问：开发中怎么选？

- **选 `String`**：如果你的字符串一旦声明后就**不再改变**，或者只是偶尔进行极少量的拼接（少量的拼接编译器会自动帮你优化）。

- **选 `StringBuilder`**：如果你需要在**单线程**环境下进行**大量或频繁**的字符串拼接、反转、替换等操作（这是**最常见**的场景，强烈推荐）。

- **选 `StringBuffer`**：如果你必须在**多线程**环境下并发修改同一个字符串缓冲区（实际开发中这种场景相对较少，现代并发编程更倾向于其他机制）。

### 三、面向对象的三大特性是什么

#### 1.封装

把对象的属性和行为封装在类里面，对外只暴露必要的方法

隐藏内部细节，提高安全性和可维护性

#### 2.继承

子类可以继承父类的属性和方法，实现代码复用

#### 3.多态

同一个父类引用，指向不同子类对象时，调用同一个方法会表现出不同的行为

```java
class Animal {
    public void speak() {
        System.out.println("动物叫");
    }
}

class Dog extends Animal {
    public void speak() {
        System.out.println("狗叫");
    }
}

class Cat extends Animal {
    public void speak() {
        System.out.println("猫叫");
    }
}

Animal a1 = new Dog();
Animal a2 = new Cat();

a1.speak(); // 狗叫
a2.speak(); // 猫叫
```

父类引用指向子类对象，方法调用看实际对象

### 四、抽象类和接口的区别，JDK8之后呢？

在JDK8之前，它们的界限非常清晰：抽象类可以有实现代码和状态（变量），而接口纯粹是抽象的方法生命，但自JDK8引入了default方法和static方法之后，接口也可以包含具体的实现代码，它们的区别就变得有些模糊。

尽管JDK8缩小了它们在语法上的差距，但它们在设计哲学上的初衷依然不同：

- **抽象类表示 “Is-A”（是一个）的关系：** 抽象类是对事物本质的抽象。它通常用于提炼出一组紧密相关的类的**公共特性和代码**，作为一个类的层级结构的基类。例如：`Dog` 和 `Cat` 都是 `Animal`（动物是抽象类）。

- **接口表示 “Can-Do”（能做）或 “Has-A”（具备）的契约：** 接口是对行为的抽象。它定义了一组规范，告诉外界“实现了我的类就一定能做这些事”，而不关心这些类在本质上是什么。例如：`Bird` 和 `Airplane` 都能 `Flyable`（可飞行是接口），但它们本质上毫无关联。

在JDK8之前，向已有接口添加新方法是一场灾难，因为所有实现了这个接口的类都会报错，必须强制实现新方法。为了在不破坏现有代码的情况下扩展接口，JDK8允许在接口中定义包含具体实现的方法：

**`default` 方法：** 允许接口提供方法的默认实现。实现类可以直接继承该方法，也可以重写它。

**`static` 方法：** 允许在接口中定义可以通过接口名直接调用的静态辅助方法。（本质与类的静态方法类似，通过类名.方法（）调用）

### 五、重载和重写的区别

重载是在同一个类中函数名相同，参数列表不同的方法；重写是子类对父类同名同参数方法的改写

因为编译器只需要看方法名、参数个数、参数顺序、参数类型就能决定调用哪个方法，所以重载也叫编译器多态

重写是在运行阶段才确定真正调用哪个方法，因此叫运行期多态

还有一个点是重写时子类不能缩小父类方法的访问权限

```
public > protected > 默认权限 > private
```

### 六、Java异常体系

#### 1.Error和Exception的区别

它们都继承自throwable，但语义完全不同

**Error**通常表示系统级、JVM级的问题，比如

- `OutOfMemoryError`

- `StackOverflowError`

这类问题一般不是业务代码能优雅恢复的，所以不建议捕获后继续运行。

**Exception**则更多表示程序运行过程中的异常情况，往往是可以通过代码处理的，比如：

- 文件不存在

- 参数非法

- 空指针

- 数组越界

#### 2.受检异常和运行时异常

**受检异常**继承自Exception类，编译器会强制检查这类异常。如果你的代码中调用了可能抛出受检异常的方法，你必须显式地使用try-catch块捕获它，或者在方法签名中使用throws关键字向上抛出，例`IOException`（文件读写失败）、`SQLException`（数据库操作失败）、`ClassNotFoundException`。

**非受检异常**继承自RuntimeException，编译器不会强制要求处理这类异常，它们通常在程序运行期间由于逻辑错误而发生。例`NullPointerException`（空指针）、`IndexOutOfBoundsException`（数组越界）、`IllegalArgumentException`（参数非法）。

Java创造者设计受检异常的初衷是为了区分“环境问题”与“程序bug”：

**环境问题（受检异常）**：比如网络突然断开、要读取的文件不存在。这些异常是程序外部的、开发者无法控制的客观因素，Java希望通过强制捕获，逼迫开发者在写代码时就像好兜底方案，这是一种防御性编程的体现。

**程序Bug（非受检异常）**：比如空指针、数组越界。这些纯粹是程序员因失误引入的Bug，是可以在测试阶段避免的。此时不需要强制捕获，而是应该让程序直接崩溃（Fail-Fast），并在日志中暴露错误，让开发者去修复代码逻辑。

但理想很美好，在长期的工业级工程实践中，受检异常暴露了一些明显的弊端：

在真实的业务场景中，绝大多数的受检异常是无法恢复的，比如当你调用底层JDBC抛出`SQLException`（比如数据库服务器宕机了）时，处于业务逻辑层的代码根本无能为力。强制要求捕获它只会导致无意义的抛出传递

随着 Spring 等现代框架的崛起，业界形成了一个默契：**尽量使用非受检异常（RuntimeException）**。

- **Spring 的做法：** Spring 会将底层繁杂的受检异常（如 JDBC 的 `SQLException`）捕获，并统一包装、转换成 Spring 自己定义的非受检异常（如 `DataAccessException`）再抛出。这样业务代码就不必再写繁琐的 `try-catch`。

- **全局异常处理：** 现在的标准做法是将这些 `RuntimeException` 一直向上抛，抛到 Web 层，由全局异常处理器（如 `@ControllerAdvice`）统一拦截，记录日志，并向前端返回标准的 JSON 错误码（如 HTTP 500 或自定义业务状态码）。

### 七、final、finally、finalize分别干什么？

| 名称           | 类型             | 作用                           |
| ------------ | -------------- | ---------------------------- |
| `final`      | 关键字            | 修饰变量、方法、类，表示“不可改变/不可重写/不可继承” |
| `finally`    | 异常处理代码块        | 无论是否发生异常，通常都会执行，用于释放资源       |
| `finalize()` | `Object` 类中的方法 | 对象被 GC 回收前可能调用，但已经不推荐使用      |

final修饰基本变量就是不能再更改值；修饰引用类型地址不能变，但对象内容可以变；修饰方法不能被子类重写；修饰类不能被继承。

finally一般是在finally块中做一些资源释放的操作，要注意的是别在finally块中写return，这会覆盖`try/catch` 里的返回值，面试和实际开发都算坑。

`finalize()` 是 `Object` 类里的一个方法，早期设计是：对象被垃圾回收之前，JVM 可能会调用它做清理工作，但重点是不要依赖它，有三个原因:

- 不确定什么时候执行；
- 不保证一定执行；
- 性能和安全性都有问题。

### 八、深拷贝和浅拷贝

#### 区别

关键词：**引用地址**VS**实际对象**

这两者的核心区别在于对**引用类型**成员变量的处理方式不同。

- 浅拷贝

它只复制对象本身，对于对象内部的引用类型变量（比如对象里有个List），它只复制引用的地址，不复制引用的对象。这样两个对象的内的引用变量指向堆内存中的同一个实体，修改其中一个，另外一个就跟着变，Java中默认的Object.clone()方法就是浅拷贝。

- 浅拷贝

它不仅复制对象本身，还会递归地复制对象内部所有引用指向的对象，这样原对象和新对象在内存中就是完全独立的，修改其中一个，互不影响。



#### 如何实现深拷贝

理论上可以通过重写clone()方法，在里面手动对每个引用对象再次调用clone来实现，但这在实际开发中非常不推荐。因为代码太繁琐，容易漏写，且维护成本高。

在工程实践中，我通常使用序列化来实现深拷贝。具体有两种方式：

**1.JSON序列化（推荐）**

这是最常用的方法，利用Jackson或FastJson把对象转成JSON字符串，再转回Java对象，如：`User copy = JSON.parseObject(JSON.toJSONString(original), User.class);` 优点是代码简单，通用性强

**2.Java原生序列化**

让对象实现`Serializable`接口，然后通过字节流写出去再读回来，这种方式比较重，但在处理一些不支持JSON转换的复杂对象时（比如对象里有循环引用）会用到。
