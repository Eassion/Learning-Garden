### 一、创建线程有几种方式？推荐哪种？

常见的创建线程方式主要有四种：

**第一种，继承 Thread 类。** 
直接重写 `run()` 方法，简单直观，但耦合度高，不够灵活。

```java
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("继承 Thread 创建线程：" + Thread.currentThread().getName());
    }
}

public class ThreadTest {
    public static void main(String[] args) {
        MyThread t = new MyThread();
        t.start(); // 启动线程
    }
}
```

**第二种，实现 Runnable 接口。** 
这是更常见的方式，因为它把“任务”和“线程”解耦了。

```java
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("实现 Runnable 创建线程：" + Thread.currentThread().getName());
    }
}

public class RunnableTest {
    public static void main(String[] args) {
        Thread t = new Thread(new MyRunnable());
        t.start();
    }
}
```

**第三种，实现 Callable 接口配合 FutureTask。** 
它适合有返回值、还能抛异常的异步任务。

```java
import java.util.concurrent.Callable;
import java.util.concurrent.FutureTask;

class MyCallable implements Callable<Integer> {
    @Override
    public Integer call() throws Exception {
        return 1 + 2;
    }
}

public class CallableTest {
    public static void main(String[] args) throws Exception {
        FutureTask<Integer> task = new FutureTask<>(new MyCallable());
        Thread t = new Thread(task);
        t.start();

        Integer result = task.get(); // 获取返回值
        System.out.println("Callable 返回结果：" + result);
    }
}
```

**第四种，通过线程池创建线程。** 
这是生产环境最推荐的方式，因为线程池可以复用线程、统一管理资源。

如果面试官问项目里怎么用，我会直接回答：**生产环境优先线程池，不会手动频繁 new Thread。**

### 三、synchronized底层原理是什么？

#### a.一个Java对象在内存中由哪几部分组成？

在 HotSpot 虚拟机中，对象在内存中的布局可以分为三块区域：

1. **对象头【核心】**：它包含两部分信息。
- **Mark Word（标记字段）**：存储对象自身的运行时数据，如 **HashCode**、**GC 分代年龄**、**锁状态标志**（无锁/偏向锁/轻量级锁/重量级锁）、线程持有的锁等。
  
  <mark>在重量级锁状态（锁标志位10）时，这个字段被改写为指向Monitor的指针</mark>

   不同锁状态下 Mark Word 存什么

| 锁状态   | Mark Word 主要存什么                       | 标志位   |
| ----- | ------------------------------------- | ----- |
| 无锁    | 对象的 `hashCode`、GC 分代年龄 `age`、是否偏向锁标志等 | `01`  |
| 可偏向状态 | 还没偏向具体线程，存 `thread = 0`、epoch、age 等   | `101` |
| 偏向锁   | 偏向的线程 ID / 线程指针、epoch、age             | `101` |
| 轻量级锁  | 指向当前线程栈中 **Lock Record** 的指针          | `00`  |
| 重量级锁  | 指向堆中 **ObjectMonitor** 的指针            | `10`  |
| GC 标记 | GC 标记相关信息                             | `11`  |

- **Klass Pointer（类型指针）**：对象指向它的类元数据的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例。

- (如果是数组，还有一块用于记录数组长度)
2. **实例数据**：这是对象真正存储的有效信息，也就是我们在代码里定义的各种成员变量（字段）的内容。

3. **对齐填充**：这不是必须的，仅仅起占位符的作用。因为 HotSpot 规定对象的起始地址必须是 **8 字节的整数倍**。如果实例数据不够 8 字节，就需要填充对齐。

#### b、Monitor结构

```
Monitor
├── Owner：当前持有锁的线程
├── EntryList：等待获取锁的线程队列（获取锁失败的线程就进入这个队列）
└── WaitSet：调用 wait() 后等待被唤醒的线程集合
（调用wait后会发生两件事，1.当前线程释放锁 2.当前线程进入这个对象Monitor的waitSet
之后需要其他线程调用notify()或notifyAll()唤醒线程）
```

#### c、synchronized锁的是什么？

对于普通同步方法

```java
public synchronized void add() {
    count++;
}
```

等价于锁住当前对象：

```java
public void add() {
    synchronized (this) {
        count++;
    }
}
```

这里的Monitor属于 this对象

对于静态同步方法

```java
public static synchronized void test() {
    // ...
}
```

锁住的是当前类的Class对象，类似于：

```java
synchronized (Counter.class) {
    // ...
}
```

 作用的对象是这个类的所有对象

修饰一个代码块，被修饰的代码块是同步语句块，起作用的范围是大括号{}里面的代码，作用的对象是调用这个代码块的对象

#### d、Monitor 的典型执行过程

看例子：

```java
Object lock = new Object();

Thread t1 = new Thread(() -> {
    synchronized (lock) {
        System.out.println("t1 get lock");
        try {
            Thread.sleep(3000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println("t1 release lock");
    }
});

Thread t2 = new Thread(() -> {
    synchronized (lock) {
        System.out.println("t2 get lock");
    }
});

t1.start();
t2.start();
```

执行过程大概是：

1. t1 先进入 synchronized，拿到 lock 对象的 Monitor
2. Monitor 的 Owner = t1
3. t2 也想进入 synchronized，但发现 Owner 不是空
4. t2 进入 EntryList 阻塞等待
5. t1 执行完，释放锁
6. t2 重新竞争锁
7. t2 成功进入 synchronized

#### e、Monitor 是可重入的

`synchronized` 是可重入锁。

也就是说，同一个线程已经拿到了某个对象的锁，它可以再次进入这个对象的同步代码。

```java
class Demo {
    public synchronized void a() {
        b();
    }

    public synchronized void b() {
        System.out.println("b");
    }
}
```

线程进入 `a()` 时已经拿到了 `this` 的 Monitor。

然后 `a()` 调用 `b()`，`b()` 也是 synchronized，还是要拿 `this` 的 Monitor。

如果锁不可重入，这里就会死锁。

但 Java 的 Monitor 支持可重入，所以同一个线程可以重复获取同一把锁。

底层会有一个计数器：

第一次进入，计数 +1
再次进入，计数 +1
退出一次，计数 -1
计数归零，真正释放锁

### 四、`synchronized`和`ReentrantLock`区别？

| **对比点** | **synchronized** | **ReentrantLock**                     |
| ------- | ---------------- | ------------------------------------- |
| 锁类型     | JVM 内置关键字        | JDK 显式锁（`java.util.concurrent.locks`） |
| 锁获取方式   | 自动加锁 / 解锁        | 手动 `lock()`/ `unlock()`               |
| 锁释放     | 自动（代码块结束 / 异常）   | 必须手动释放（通常在 `finally`）                 |
| 可重入性    | ✅ 可重入            | ✅ 可重入                                 |
| 公平性     | ❌ 非公平            | ✅ 支持公平 / 非公平                          |
| 可中断锁    | ❌                | ✅ `lockInterruptibly()`               |
| 尝试获取锁   | ❌                | ✅ `tryLock()`                         |
| 条件队列    | 单一 `wait/notify` | 多个 `Condition`                        |
| 性能      | JDK 1.6 后大幅优化    | 高并发下更稳定                               |
| 使用复杂度   | 简单               | 相对复杂                                  |

### 五、volatile关键字

#### a、**什么是可见性问题？为什么会出现？**

可见性问题是指一个线程修改了共享变量的值，其他线程无法立即看到这个修改。

出现这个问题的原因是，现代计算机为了提高性能，每个 CPU 核心都有自己的缓存。线程在读取变量时，会先从自己的工作内存（缓存）中读取，而不是直接从主内存读取。当一个线程修改了变量，可能只是修改了自己缓存中的值，还没有刷新到主内存，其他线程的缓存中还是旧值，就会出现可见性问题。

举个例子，比如我有一个 `boolean flag = false`，线程 A 把它改成 true，但线程 B 可能一直读到 false，导致程序出现意外的行为。给 flag 加上 volatile 关键字就能解决这个问题。

#### b、**volatile 关键字有什么作用？**

第一个是**保证可见性**。当一个线程修改了 volatile 变量的值，这个新值会立即刷新到主内存，其他线程读取这个变量时，会从主内存中读取最新值，而不是使用自己工作内存中的旧值。

第二个是**禁止指令重排序**。编译器和处理器为了优化性能，可能会对指令进行重排序，但 volatile 通过内存屏障，禁止特定类型的重排序，保证代码按照我们预期的顺序执行。
