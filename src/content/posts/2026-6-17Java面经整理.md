### 一、创建线程有几种方式？推荐哪种？

常见的创建线程方式主要有四种：

**第一种，继承 Thread 类。** 
直接重写 `run()` 方法，简单直观，但耦合度高，不够灵活。

```java
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("继承 Thread 创建线程：" + Thread.currentThread().getName());
    }
}

public class ThreadTest {
    public static void main(String[] args) {
        MyThread t = new MyThread();
        t.start(); // 启动线程
    }
}
```

**第二种，实现 Runnable 接口。** 
这是更常见的方式，因为它把“任务”和“线程”解耦了。

```java
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("实现 Runnable 创建线程：" + Thread.currentThread().getName());
    }
}

public class RunnableTest {
    public static void main(String[] args) {
        Thread t = new Thread(new MyRunnable());
        t.start();
    }
}
```

**第三种，实现 Callable 接口配合 FutureTask。** 
它适合有返回值、还能抛异常的异步任务。

```java
import java.util.concurrent.Callable;
import java.util.concurrent.FutureTask;

class MyCallable implements Callable<Integer> {
    @Override
    public Integer call() throws Exception {
        return 1 + 2;
    }
}

public class CallableTest {
    public static void main(String[] args) throws Exception {
        FutureTask<Integer> task = new FutureTask<>(new MyCallable());
        Thread t = new Thread(task);
        t.start();

        Integer result = task.get(); // 获取返回值
        System.out.println("Callable 返回结果：" + result);
    }
}
```

**第四种，通过线程池创建线程。** 
这是生产环境最推荐的方式，因为线程池可以复用线程、统一管理资源。

如果面试官问项目里怎么用，我会直接回答：**生产环境优先线程池，不会手动频繁 new Thread。**

### 三、synchronized底层原理是什么？

#### a.一个Java对象在内存中由哪几部分组成？

在 HotSpot 虚拟机中，对象在内存中的布局可以分为三块区域：

1. **对象头【核心】**：它包含两部分信息。
- **Mark Word（标记字段）**：存储对象自身的运行时数据，如 **HashCode**、**GC 分代年龄**、**锁状态标志**（无锁/偏向锁/轻量级锁/重量级锁）、线程持有的锁等。
  
  <mark>在重量级锁状态（锁标志位10）时，这个字段被改写为指向Monitor的指针</mark>

   不同锁状态下 Mark Word 存什么

| 锁状态   | Mark Word 主要存什么                       | 标志位   |
| ----- | ------------------------------------- | ----- |
| 无锁    | 对象的 `hashCode`、GC 分代年龄 `age`、是否偏向锁标志等 | `01`  |
| 可偏向状态 | 还没偏向具体线程，存 `thread = 0`、epoch、age 等   | `101` |
| 偏向锁   | 偏向的线程 ID / 线程指针、epoch、age             | `101` |
| 轻量级锁  | 指向当前线程栈中 **Lock Record** 的指针          | `00`  |
| 重量级锁  | 指向堆中 **ObjectMonitor** 的指针            | `10`  |
| GC 标记 | GC 标记相关信息                             | `11`  |

- **Klass Pointer（类型指针）**：对象指向它的类元数据的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例。

- (如果是数组，还有一块用于记录数组长度)
2. **实例数据**：这是对象真正存储的有效信息，也就是我们在代码里定义的各种成员变量（字段）的内容。

3. **对齐填充**：这不是必须的，仅仅起占位符的作用。因为 HotSpot 规定对象的起始地址必须是 **8 字节的整数倍**。如果实例数据不够 8 字节，就需要填充对齐。

#### b、Monitor结构

```
Monitor
├── Owner：当前持有锁的线程
├── EntryList：等待获取锁的线程队列（获取锁失败的线程就进入这个队列）
└── WaitSet：调用 wait() 后等待被唤醒的线程集合
（调用wait后会发生两件事，1.当前线程释放锁 2.当前线程进入这个对象Monitor的waitSet
之后需要其他线程调用notify()或notifyAll()唤醒线程）
```

#### c、synchronized锁的是什么？

对于普通同步方法

```java
public synchronized void add() {
    count++;
}
```

等价于锁住当前对象：

```java
public void add() {
    synchronized (this) {
        count++;
    }
}
```

这里的Monitor属于 this对象

对于静态同步方法

```java
public static synchronized void test() {
    // ...
}
```

锁住的是当前类的Class对象，类似于：

```java
synchronized (Counter.class) {
    // ...
}
```

 作用的对象是这个类的所有对象

修饰一个代码块，被修饰的代码块是同步语句块，起作用的范围是大括号{}里面的代码，作用的对象是调用这个代码块的对象

#### d、Monitor 的典型执行过程

看例子：

```java
Object lock = new Object();

Thread t1 = new Thread(() -> {
    synchronized (lock) {
        System.out.println("t1 get lock");
        try {
            Thread.sleep(3000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println("t1 release lock");
    }
});

Thread t2 = new Thread(() -> {
    synchronized (lock) {
        System.out.println("t2 get lock");
    }
});

t1.start();
t2.start();
```

执行过程大概是：

1. t1 先进入 synchronized，拿到 lock 对象的 Monitor
2. Monitor 的 Owner = t1
3. t2 也想进入 synchronized，但发现 Owner 不是空
4. t2 进入 EntryList 阻塞等待
5. t1 执行完，释放锁
6. t2 重新竞争锁
7. t2 成功进入 synchronized

#### e、Monitor 是可重入的

`synchronized` 是可重入锁。

也就是说，同一个线程已经拿到了某个对象的锁，它可以再次进入这个对象的同步代码。

```java
class Demo {
    public synchronized void a() {
        b();
    }

    public synchronized void b() {
        System.out.println("b");
    }
}
```

线程进入 `a()` 时已经拿到了 `this` 的 Monitor。

然后 `a()` 调用 `b()`，`b()` 也是 synchronized，还是要拿 `this` 的 Monitor。

如果锁不可重入，这里就会死锁。

但 Java 的 Monitor 支持可重入，所以同一个线程可以重复获取同一把锁。

底层会有一个计数器：

第一次进入，计数 +1
再次进入，计数 +1
退出一次，计数 -1
计数归零，真正释放锁

### 四、`synchronized`和`ReentrantLock`区别？

| **对比点** | **synchronized** | **ReentrantLock**                     |
| ------- | ---------------- | ------------------------------------- |
| 锁类型     | JVM 内置关键字        | JDK 显式锁（`java.util.concurrent.locks`） |
| 锁获取方式   | 自动加锁 / 解锁        | 手动 `lock()`/ `unlock()`               |
| 锁释放     | 自动（代码块结束 / 异常）   | 必须手动释放（通常在 `finally`）                 |
| 可重入性    | ✅ 可重入            | ✅ 可重入                                 |
| 公平性     | ❌ 非公平            | ✅ 支持公平 / 非公平                          |
| 可中断锁    | ❌                | ✅ `lockInterruptibly()`               |
| 尝试获取锁   | ❌                | ✅ `tryLock()`                         |
| 条件队列    | 单一 `wait/notify` | 多个 `Condition`                        |
| 性能      | JDK 1.6 后大幅优化    | 高并发下更稳定                               |
| 使用复杂度   | 简单               | 相对复杂                                  |

### 五、volatile关键字

#### a、**什么是可见性问题？为什么会出现？**

可见性问题是指一个线程修改了共享变量的值，其他线程无法立即看到这个修改。

出现这个问题的原因是，现代计算机为了提高性能，每个 CPU 核心都有自己的缓存。线程在读取变量时，会先从自己的工作内存（缓存）中读取，而不是直接从主内存读取。当一个线程修改了变量，可能只是修改了自己缓存中的值，还没有刷新到主内存，其他线程的缓存中还是旧值，就会出现可见性问题。

举个例子，比如我有一个 `boolean flag = false`，线程 A 把它改成 true，但线程 B 可能一直读到 false，导致程序出现意外的行为。给 flag 加上 volatile 关键字就能解决这个问题。

#### b、**volatile 关键字有什么作用？**

第一个是**保证可见性**。当一个线程修改了 volatile 变量的值，这个新值会立即刷新到主内存，其他线程读取这个变量时，会从主内存中读取最新值，而不是使用自己工作内存中的旧值。

第二个是**禁止指令重排序**。编译器和处理器为了优化性能，可能会对指令进行重排序，但 volatile 通过内存屏障，禁止特定类型的重排序，保证代码按照我们预期的顺序执行。

#### c、**什么是指令重排序？为什么要重排序？**

指令重排序是指编译器和处理器为了优化性能，可能会改变指令的执行顺序，但会保证单线程下的执行结果不变。

重排序的目的是提高 CPU 的执行效率，减少流水线停顿。但在多线程环境下，重排序可能导致程序出现意外的结果，所以在某些关键位置，我们需要用 volatile 来禁止重排序。

#### d、**volatile 是如何保证可见性和禁止重排序的？**

内存屏障

对于可见性，当写一个 volatile 变量时，JMM（Java 内存模型）会在写操作后插入一个 Store 屏障，强制将缓存中的数据刷新到主内存。当读一个 volatile 变量时，JMM 会在读操作前插入一个 Load 屏障，强制从主内存中读取最新值，使工作内存中的缓存失效。

对于禁止重排序，volatile 通过内存屏障，禁止了几种重排序：写 volatile 变量之前的操作不能重排序到写操作之后，读 volatile 变量之后的操作不能重排序到读操作之前。这样就保证了代码的执行顺序符合我们的预期。

#### e、**volatile 能保证原子性吗？为什么？**

不能保证原子性。

我举个例子，比如 `count++` 这个操作，看起来是一条语句，但实际上包含三个步骤：读取 count 的值、将 count 的值加 1、将新值写回 count。

在多线程环境下，可能出现这样的情况：线程 A 和线程 B 同时读取 count = 0，然后都计算出 1，最后都写回 1。这样执行了两次 count++，但最终 count 只等于 1，而不是期望的 2。

如果要保证原子性，我们需要使用 synchronized 或者 Lock，或者使用 AtomicInteger 这样的原子类。

#### f、**什么情况下 volatile 可以保证线程安全？**

当满足以下两个条件时，volatile 可以保证线程安全：

第一，对变量的写操作不依赖当前值，比如直接赋值 `flag = true`，而不是 `count++` 这种依赖当前值的操作。

第二，该变量不需要与其他变量共同参与不变约束。

典型的场景就是用 volatile 修饰一个布尔变量作为状态标志，或者在双重检查锁定（DCL）单例模式中使用。

#### g、**volatile 和 synchronized 有什么区别？**

首先，**作用范围不同**。volatile 只能修饰变量，而 synchronized 可以修饰方法和代码块。

其次，**原子性不同**。volatile 不能保证原子性，synchronized 可以保证原子性。但它们都能保证可见性和有序性。

第三，**性能不同**。volatile 是轻量级的同步机制，性能更好，不会阻塞线程。synchronized 是重量级的同步机制，可能会阻塞线程，但 JDK 1.6 之后做了很多优化，性能也提升了不少。

最后，**使用场景不同**。volatile 适用于一写多读的场景，比如状态标志。synchronized 适用于需要原子性的场景，比如 count++ 这种操作。

总结一下，能用 synchronized 就用 synchronized，除非只需要保证可见性，不需要原子性，那用 volatile 就够了。

#### h、**volatile 有哪些典型的应用场景？**

第一个是**状态标志**。最常见的就是用 volatile 修饰一个布尔变量，作为线程的停止标志。比如我有一个 `volatile boolean stop = false`，在线程的 run 方法里用 `while (!stop)` 循环执行任务，当其他线程调用 `stop = true` 时，执行任务的线程能立即看到这个修改，退出循环。

第二个是**双重检查锁定（DCL）单例模式**。在单例模式中，我们会用 `volatile` 修饰 instance 变量。这是因为 `instance = new Singleton()` 这行代码实际上包含三个步骤：分配内存空间、初始化对象、将 instance 指向分配的内存地址。由于指令重排序，可能变成先分配内存、再指向内存、最后初始化对象。如果线程 A 执行到指向内存这一步时，线程 B 进入第一次检查，发现 instance 不为 null，直接返回，但此时对象还没有初始化完成，就会出现空指针异常。volatile 的作用就是禁止这种重排序，保证对象初始化完成后，才将引用赋值给 instance。

### 六、线程池

### a、**为什么要使用线程池？有什么好处？**

第一个是**降低资源消耗**。因为线程的创建和销毁是有开销的，需要调用操作系统内核。线程池通过复用已创建的线程，避免了频繁创建销毁的开销。

第二个是**提高响应速度**。任务到达时，不需要等待线程创建，可以立即执行。

第三个是**提高线程的可管理性**。线程是稀缺资源，如果无限制创建，会消耗系统资源，甚至导致系统崩溃。线程池可以统一分配、调优和监控线程。

#### b、**ThreadPoolExecutor 有哪些核心参数？分别是什么作用？**

ThreadPoolExecutor 有**七个核心参数**：

我详细说一下这七个参数：

第一个是 **corePoolSize（核心线程数）**，它表示线程池中常驻的线程数量。即使这些线程处于空闲状态，也不会被回收，除非设置了 `allowCoreThreadTimeOut`。

第二个是 **maximumPoolSize（最大线程数）**，它表示线程池允许创建的最大线程数量。当任务队列满了，且当前线程数小于最大线程数时，会创建新线程处理任务。

第三个是 **keepAliveTime（空闲线程存活时间）**，当线程数超过核心线程数时，多余的空闲线程在被回收前的最大等待时间。

第四个是 **unit（时间单位）**，就是 keepAliveTime 的时间单位，比如 `TimeUnit.SECONDS`。

第五个是 **workQueue（任务队列）**，用于存放等待执行的任务。常见的有 `ArrayBlockingQueue`（有界队列）、`LinkedBlockingQueue`（无界队列，默认容量是 Integer.MAX_VALUE）、还有 `SynchronousQueue`（不存储元素的队列，每个插入操作必须等待一个移除操作）。

第六个是 **threadFactory（线程工厂）**，用于创建新线程。我们可以自定义线程名称、优先级等。

第七个是 **handler（拒绝策略）**，当任务队列满了，且线程数达到最大值时，如何处理新提交的任务。

#### c、**线程池的执行流程是怎样的？**

首先，我们调用 `execute()` 或 `submit()` 提交任务。

然后，线程池会判断当前运行的线程数是不是小于核心线程数。如果小于，就直接创建新线程执行任务，即使有空闲线程也会创建新的。

如果当前运行的线程数大于等于核心线程数，任务会被放入任务队列等待。

如果任务队列已满，且当前线程数小于最大线程数，就会创建新的非核心线程执行任务。

最后，如果任务队列满了，且线程数已达到最大值，就会执行拒绝策略。

**为什么线程数小于核心线程数时，即使有空闲线程也要创建新线程？**

这是线程池的设计策略。核心线程是"常驻"的，线程池会尽快达到核心线程数，以保证有足够的线程处理任务。这样可以避免任务在队列中等待，提高响应速度。

#### d、**线程池有哪些拒绝策略？分别适用什么场景？**

第一个是 **AbortPolicy（默认）**，它会直接抛出异常 `RejectedExecutionException`。这个适用于需要感知任务被拒绝的情况，及时发现问题。

第二个是 **CallerRunsPolicy**，它会由调用线程（提交任务的线程）自己执行任务。这个适用于任务不能丢失，且可以接受降低提交速度的场景。这种策略会降低任务提交的速度，起到一定的限流作用。

第三个是 **DiscardPolicy**，它会直接丢弃任务，不抛异常。这个适用于允许任务丢失的场景，比如日志记录。

第四个是 **DiscardOldestPolicy**，它会丢弃队列中最老的任务，然后尝试重新提交当前任务。这个适用于优先处理新任务的场景。

**生产环境中你会选择哪种拒绝策略？**

这要看具体业务场景：

如果是**核心业务**，比如订单、支付这种，我会使用 **AbortPolicy**，抛异常后可以记录日志、告警，不能让任务悄悄丢失。

如果是**非核心业务**，比如日志、埋点这种，可以用 **DiscardPolicy**，允许部分数据丢失。

如果是**需要限流**的场景，我会用 **CallerRunsPolicy**，让调用线程自己执行，降低提交速度。

#### e、**Executors 提供了哪些常见的线程池？有什么问题？**

第一个是 **newFixedThreadPool(int n)**，它是固定大小的线程池，核心线程数等于最大线程数等于 n。它使用 `LinkedBlockingQueue`（无界队列）。问题是队列无界，可能导致内存溢出（OOM）。

第二个是 **newSingleThreadExecutor()**，它只有一个线程的线程池，也使用 `LinkedBlockingQueue`。同样存在 OOM 风险。

第三个是 **newCachedThreadPool()**，它的核心线程数为 0，最大线程数为 `Integer.MAX_VALUE`，使用 `SynchronousQueue`（不存储任务）。问题是可能创建大量线程，导致系统资源耗尽。

第四个是 **newScheduledThreadPool(int n)**，它支持定时和周期性任务执行。同样使用无界队列，存在 OOM 风险。

**为什么阿里巴巴开发手册不推荐使用 Executors 创建线程池？**

主要有两个原因：

第一个是 **FixedThreadPool 和 SingleThreadExecutor** 使用的是 `LinkedBlockingQueue`，默认容量是 `Integer.MAX_VALUE`（约 21 亿）。如果任务提交速度大于处理速度，队列会无限增长，导致内存溢出。

第二个是 **CachedThreadPool** 的最大线程数是 `Integer.MAX_VALUE`，可能创建大量线程，导致 CPU 和内存资源耗尽。

所以推荐的做法是使用 `ThreadPoolExecutor` 手动创建线程池，明确指定核心参数，避免资源失控。比如这样：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,                          // 核心线程数
    20,                          // 最大线程数
    60L,                         // 空闲线程存活时间
    TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100),  // 有界队列
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);
```

#### f、**如何合理设置线程池的核心线程数和最大线程数？**

线程池参数的配置需要根据**任务类型**来决定：

**第一种是 CPU 密集型任务**，任务主要消耗 CPU 资源，比如计算、加密这种。推荐配置是核心线程数等于 CPU 核心数加 1。原因是 CPU 密集型任务，线程数过多会导致频繁的上下文切换，反而降低性能。加 1 是为了在某个线程因为偶尔的内存页失效或其他原因暂停时，额外的线程可以顶上。

**第二种是 IO 密集型任务**，任务主要等待 IO 操作，比如网络请求、数据库查询、文件读写这种。推荐配置是核心线程数等于 CPU 核心数乘以 2，或者 CPU 核心数除以（1 减去阻塞系数）。阻塞系数通常在 0.8 到 0.9 之间，也就是线程有 80% 到 90% 的时间在等待 IO。原因是 IO 密集型任务，线程大部分时间在等待，可以创建更多线程来提高 CPU 利用率。

**第三种是混合型任务**，建议拆分成 CPU 密集型和 IO 密集型任务，分别用不同的线程池处理。



**你在项目中是怎么配置线程池的？**

在我的项目中，我会根据实际业务场景来配置：

比如**异步任务处理**，像发送短信、邮件这种，这是典型的 IO 密集型任务。我会设置核心线程数为 20，最大线程数为 50，使用有界队列（比如 `ArrayBlockingQueue(200)`），拒绝策略用 `CallerRunsPolicy`。

如果是**数据计算任务**，这是 CPU 密集型任务。我会设置核心线程数等于 CPU 核心数加 1，最大线程数与核心线程数相同，使用较小的队列。

重点是，生产环境中，我会通过压测和监控（比如线程池的活跃线程数、队列大小、拒绝次数）来动态调整参数，找到最优配置。

#### g、**如何正确关闭线程池？shutdown 和 shutdownNow 有什么区别？**

线程池提供了两种关闭方法：

第一个是 **shutdown()**，这是温柔关闭。它不再接受新任务，但会执行完队列中的所有任务。已提交的任务会继续执行，线程池状态变为 `SHUTDOWN`。

第二个是 **shutdownNow()**，这是强制关闭。它不再接受新任务，尝试停止正在执行的任务，并返回队列中未执行的任务列表。它通过调用线程的 `interrupt()` 方法来中断线程，线程池状态变为 `STOP`。

**为什么不能直接用 shutdownNow？**

因为 `shutdownNow()` 只是尝试中断线程，但如果任务没有正确处理中断信号（比如没有检查 `Thread.interrupted()`），任务可能无法停止。而且，正在执行的任务可能处理到一半，导致数据不一致。所以生产环境中，优先使用 `shutdown()` 优雅关闭。

#### h、`execute()` **和** `submit()` **有什么区别？**

`execute()` 只提交任务，没有返回值；`submit()` 会返回 `Future`，可以拿结果，也会把异常包装到 `Future` 里。面试里一般会补一句：如果要感知异步任务异常，不能只调 `submit()` 不看 `Future`。
