第一步是**抓取**现场，通过慢查询日志或者 `SHOW PROCESSLIST` 把有问题的 SQL 揪出来。
第二步是**分析** SQL 本身，这也是最常用的一步。我会用 `EXPLAIN` 查看它的执行计划，看是否走了索引、有没有触发文件排序。如果计划没问题，我会结合 `SHOW PROFILES` 查看具体的耗时环节。
第三步，如果语句本身没问题，我会排查**锁**等待，看看是不是有长事务阻塞了当前查询。
最后，如果数据库层面都没异常，我会排查宿主**机器**的 CPU 或磁盘 IO 是否达到了硬件瓶颈。 

##### 1. 抓：抓取慢 SQL（找案发现场）

- **日常排查：** 开启**慢查询日志** (`slow_query_log`)，配合 `mysqldumpslow` 或 `pt-query-digest` 工具，找出历史高频且耗时的 SQL。

- **线上突发卡顿：** 直接执行 `SHOW PROCESSLIST;`，抓出当前正在执行且耗时极长的异常线程。

##### 2. 析：分析 SQL 本身（找内因，这是重头戏）

- **第一手武器：`EXPLAIN` 分析执行计划。** * 重点看索引是否命中（`key` 字段）。
  
  - 看扫描级别（`type` 字段，是全表扫描 `ALL` 还是索引扫描 `ref`/`range`）。
  
  - 看额外信息（`Extra` 字段，有没有出现致命的 `Using filesort` 内存排序或 `Using temporary` 临时表）。

- **第二手武器：`SHOW PROFILES` 剖析耗时。** * 如果 `EXPLAIN` 看着没问题（索引走得很好），那就用它看看时间到底耗在了哪个阶段（是网络传输慢，还是生成临时表慢）。

##### 3. 锁：排查锁阻塞（找外因）

- **核心逻辑：** 很多时候 SQL 写得很好，只是被其他事务的锁“堵”住了。

- **排查手段：** 通过查询 `sys.innodb_lock_waits` 系统视图，精准定位是谁在等锁，又是哪个长事务占着锁不释放。

##### 4. 机：排查机器瓶颈（看大环境）

- **核心逻辑：** 数据库服务器底层的资源见底了。

- **关注指标：** 检查操作系统的 CPU 是否飙升、磁盘 IOPS 是否打满，或者网络宽带是否拥塞。

##### 5.解决问题

确认问题后一般通过新建或调整联合索引、避免索引失效写法如对字段做函数运算、优化关联查询和分页、或减少SELECT *及返回数据量来进行优化，同时在测试环境验证执行计划是否改善。

#### 如果让你创建联合索引，你会怎么做？

如果让我创建联合索引，我会结合具体的业务 SQL 来定。首先我会遵循**最左前缀法则**，把高频条件放最左边；其次我会计算字段的**区分度**，高区分度的优先；同时我会注意把**范围查询的字段尽量靠后放**，以免阻断索引；最后，如果条件允许，我会尽量涵盖 SELECT 需要的字段，利用**覆盖索引**来避免回表。

计算字段区分度：

```sql
SELECT COUNT(DISTINCT column_name) / COUNT(*) AS selectivity FROM table_name;
```

#### 索引实效的场景有哪些？

1.对索引使用函数
SELECT * FROM orders WHERE DATE(create_time) = '2026-06-24';
2.对索引使用表达式计算
SELECT * FROM orders WHERE score + 10 = 100;
3.存在索引隐式类型转换

假设user_id在表中定义的是varchar，但在java代码里或者写sql时传了数字，MySQL 会在底层**自动加函数**帮你做类型转换
SELECT * FROM orders WHERE user_id = 123456;
4.联合索引违反最左前缀原则
（缺左）SELECT * FROM orders WHERE status = 1;
（断链）SELECT * FROM orders WHERE create_time > '2026-06-01' AND status = 1;

（模糊）LIKE '%张三' 或 LIKE '%张三%'

**容易忘记：**

如果 `WHERE` 条件里用了 `OR`，那么 `OR` 连接的**所有字段都必须有单独的索引**，否则整个查询都会退化为全表扫描。

- **场景：** `WHERE id = 1 OR age = 18`。假设 `id` 有索引，但 `age` 没有索引。

- **结果：** 索引失效。因为即使 `id` 走索引查得快，`age` 还是得扫全表去验证，MySQL 优化器一盘算：还不如直接全表扫描算了一次性搞定。

- **原因：** 需要先索引找id，再全表找age，多此一举，直接全表，所以造成索引失效

继续问：OR两边都有单列索引，还会失效吗？

MySQL 5.0 以后的版本中，引入了索引合并（Index Merge）技术，会先去id索引找符合条件的主键id，再去age索引找符合条件的主键id，然后在内存里边并集去重，然后再去聚簇索引捞完整数据，这时候你如果去用 `EXPLAIN` 查这条语句，你会看到非常有趣的现象：

- **`type` 字段：** 显示为 **`index_merge`**（而不是平时看到的 `ref` 或 `range`）。

- **`Extra` 字段：** 会显示 **`Using union(name_index, age_index)`**。



#### ORDER BY 和 GROUP BY 也遵守最左前缀原则吗？

##### 1. `ORDER BY` 与最左前缀（核心目的：消灭 `filesort`）

B+ 树本身就是一棵排好序的树。如果你的 `ORDER BY` 顺应了这棵树的固有排序规则，MySQL 只需要**顺藤摸瓜**（按叶子节点链表顺序读取），连一丁点额外的排序操作都不用做。

假设我们有联合索引 `(A, B, C)`：

- **✅ 完美利用索引（顺着树读）：**
  
  - `ORDER BY A`
  
  - `ORDER BY A, B`
  
  - `ORDER BY A, B, C`
  
  - *底层逻辑：这完全契合 B+ 树先按 A 排、再按 B 排的规则，`EXPLAIN` 的 `Extra` 字段极其干净。*

- **❌ 违反最左前缀（引发 `Using filesort`）：**
  
  - `ORDER BY B, C`（跳过了 A）
  
  - `ORDER BY A, C`（跳过了 B，只有 A 能用索引排序）
  
  - *底层逻辑：因为跳过了前置字段，剩下的字段在整棵树上是**无序**的。MySQL 无奈，只能把数据全都查出来，扔进内存的 Sort Buffer 里重新进行一次排序（如果内存不够，还要借助磁盘文件排序）。在 `EXPLAIN` 里，这就是极其碍眼的 `Using filesort`。*

**⚠️ 面试必考的“变种题”：** 面试官常常会把 `WHERE` 和 `ORDER BY` 结合起来考。
比如：`SELECT * FROM table WHERE A = 1 ORDER BY B, C;` **这条语句走不走索引排序？** **答案是：走！** 因为 `WHERE A = 1` 已经把 A 确定为一个固定值了。在 A 确定的“局部空间”里，B 和 C 本来就是绝对有序的。这依然是完美符合最左前缀原则的变形应用。

##### 2. `GROUP BY` 与最左前缀（核心目的：消灭 `temporary`）

理解了 `ORDER BY`，`GROUP BY` 简直就是白送的。
你思考一下，要在上百万条数据里，把相同的值“归类”到一起（分组），最笨的办法是什么？是建一个临时表，扫一条对一条。最聪明的办法是什么？是**先把数据排好序，这样相同的值自然就紧挨在一起了，扫一遍就能轻松分好组**。

也就是说：**`GROUP BY` 的底层几乎总是伴随着排序动作。**

假设依然是联合索引 `(A, B, C)`：

- **✅ 完美利用索引：**
  
  - `GROUP BY A, B`
  
  - *底层逻辑：数据已经在 B+ 树上按 A 和 B 扎堆聚在一起了，MySQL 只需要顺着查，遇到不同的值就划条线分个组，极其高效。*

- **❌ 违反最左前缀（引发 `Using temporary; Using filesort`）：**
  
  - `GROUP BY B`（跳过了 A）
  
  - *底层逻辑：B 的值在树上散落得到处都是。MySQL 没辙了，只能先建一张**临时表**（`Using temporary`），把数据全丢进去，然后再对临时表进行**排序**（`Using filesort`），最后才能完成分组。这俩兄弟同时出现在 `EXPLAIN` 里，堪称性能灾难。*
