严格来说，现在常说的 SSL，其实大多数场景指的是 TLS

SSL是早期协议，已经不安全，基本被淘汰。TLS就是SSL 的后续版本，现在 HTTPS 主要用 TLS

TLS 主要用来保证通信的机密性、完整性和身份认证。它解决三个问题：

```
1. 防窃听：别人抓包也看不懂内容
2. 防篡改：数据被改了能发现
3. 防伪装：确认访问的服务器是真的，不是假的
```

它主要做三件事：

```
1. 协商加密参数
2. 验证服务器身份
3. 生成对称加密密钥
```

HTTPS 在 TCP 三次握手之后，会进行 TLS 握手。TLS 握手的主要目的，是协商加密参数、验证服务器身份，并生成后续通信使用的会话密钥。

首先客户端发送 ClientHello，里面包含客户端支持的 TLS 版本、加密套件、随机数等信息。服务器收到后返回 ServerHello，选择一个双方都支持的 TLS 版本和加密套件，并返回服务器随机数。然后服务器会发送数字证书，证书中包含服务器公钥、域名、CA 签名等信息。

浏览器收到证书后，会验证证书是否由可信 CA 签发、是否过期、域名是否匹配、证书链是否可信等。如果证书验证通过，客户端和服务器会进行密钥协商。现代 TLS 通常使用 ECDHE，双方交换临时公钥，并各自计算出相同的共享密钥，再结合随机数生成最终的会话密钥。

密钥生成后，双方会发送 Finished 消息，确认握手过程没有被篡改，并验证双方是否拥有相同的密钥。TLS 握手完成后，后续的 HTTP 请求和响应都会使用这个会话密钥进行对称加密传输。

所以 TLS 的核心可以概括为：先用证书验证服务器身份，再通过密钥交换算法安全生成会话密钥，最后用对称加密传输真正的数据。

```
客户端                                      服务器
  |                                          |
  | ------------ TCP 三次握手 -------------->  |
  |                                           |
  | ----------- ClientHello ----------------> |
  |      支持的 TLS 版本、加密套件、随机数         |
  |                                           |
  | <---------- ServerHello ----------------- |
  |       选择的 TLS 版本、加密套件、随机数        |
  |                                           |
  | <---------- Certificate ----------------- |
  |        服务器证书，包含服务器公钥              |
  |                                           |
  | <---------- ServerKeyExchange ----------- |
  |        密钥交换参数，比如 ECDHE 公钥          |
  |                                           |
  | ----------- ClientKeyExchange ----------> |
  |             客户端密钥交换参数               |
  |                                           |
  |           双方计算出相同的会话密钥            |
  |                                           |
  | ----------- Finished -------------------> |
  | <---------- Finished -------------------- |
  |                                           |
  | -------- 加密 HTTP 请求 ---------------->  |
  | <------- 加密 HTTP 响应 -----------------  |
```

### 密钥协商过程

证书验证通过后，客户端和服务器要协商出一个共同的会话密钥。

重点说两种：RSA 和 ECDHE。

#### 1. 早期 RSA 密钥交换

早期 TLS 可能使用 RSA 密钥交换。

大致流程是：

```
1. 服务器把证书发给客户端
2. 证书中包含服务器公钥
3. 客户端生成一个随机数，叫 pre-master secret
4. 客户端用服务器公钥加密这个 pre-master secret
5. 服务器用自己的私钥解密，得到 pre-master secret
6. 双方根据 Client Random、Server Random、pre-master secret 生成会话密钥
```

这个流程的关键是：

```
只有服务器有私钥，所以只有服务器能解密 pre-master secret。
```

但是 RSA 密钥交换有一个问题：如果服务器私钥以后泄露了，攻击者之前抓到的历史 TLS 通信，也可能被解密。这叫不具备前向安全性。

#### 2. 现在更常见的是 ECDHE

现代 TLS 更常用 ECDHE 做密钥交换。

ECDHE 可以实现前向安全性。

大致流程是：

```
1. 客户端生成一对临时密钥
2. 服务器也生成一对临时密钥
3. 双方交换各自的临时公钥
4. 双方根据自己的临时私钥和对方的临时公钥，计算出同一个共享密钥
5. 再结合 Client Random、Server Random，生成最终会话密钥
```

这个过程有一个特点：双方不需要直接传输真正的会话密钥。

即使中间人抓到了网络包，也算不出最终密钥。

而且这些密钥是临时生成的，会话结束后就丢弃。

所以即使服务器长期私钥以后泄露，之前的通信内容也不容易被解密。

这就是：前向安全性

### 为什么不直接用非对称加密传输数据？

因为非对称加密比较慢。

比如 RSA、公钥、私钥这类属于非对称加密：

```
优点：安全，适合密钥交换和身份验证缺点：性能较差，不适合加密大量数据
```

而 AES 这类属于对称加密：

```
优点：速度快，适合加密大量数据
缺点：需要双方提前拥有同一个密钥
```

所以 TLS 的设计思路是：

```
握手阶段：用非对称加密 / 密钥交换算法，安全地协商密钥
传输阶段：用对称加密，快速加密 HTTP 数据
```

### 浏览器怎么验证证书链？

```
服务器证书：www.example.com
Issuer: Intermediate CA
        ↓
用 Intermediate CA 的公钥验证服务器证书签名
        ↓
中间 CA 证书：Intermediate CA
Issuer: Root CA
        ↓
用 Root CA 的公钥验证中间 CA 证书签名
        ↓
根 CA 证书：Root CA
        ↓
检查 Root CA 是否在浏览器/操作系统可信根证书库中
        ↓
如果在，证书链可信
```

服务器证书里面有两个重要字段：

Subject：这个证书属于谁
Issuer：这个证书是谁签发的

浏览器验证过程：

1. 用中间 CA 的公钥验证服务器证书签名
2. 用根 CA 的公钥验证中间 CA 证书签名
3. 检查根 CA 是否在本机可信根证书库里

根 CA 证书通常是自签名证书，也就是：

Subject = Root CA
Issuer = Root CA

浏览器会检查这个根 CA 是否在自己的信任列表里
