# Redis 分布式锁核心实现原理

在单机应用中，一把 `synchronized` 或 `ReentrantLock` 就能解决并发问题。但在分布式系统中，多个服务实例运行在不同的 JVM 中，本地锁已经无法保证互斥，因此需要**分布式锁**。

Redis 因其性能高、支持原子操作，被广泛用于实现分布式锁。

---

# 一、分布式锁需要解决什么问题？

一个合格的 Redis 分布式锁，至少需要满足以下几点：

- **互斥性**：同一时刻只有一个客户端能够获取锁。
- **防死锁**：客户端异常退出后，锁能够自动释放。
- **安全释放**：只能释放自己持有的锁，不能误删其他客户端的锁。
- **高可用**：Redis 故障时尽量保证锁的可靠性（可选）。

---

# 二、基础方案：SETNX + EXPIRE（存在缺陷）

最早的实现方式如下：

```redis
SETNX lock_key value
EXPIRE lock_key 30
```

执行流程：

1. `SETNX`：如果 Key 不存在，则获取锁成功。
2. `EXPIRE`：为锁设置过期时间，避免死锁。

这种方案的问题在于：**两条命令不是原子操作。**

如果执行完 `SETNX` 后服务宕机，而 `EXPIRE` 还未来得及执行，锁将永远不会过期，从而造成死锁。

---

# 三、官方推荐方案：SET NX EX

Redis 官方推荐使用一条命令完成加锁：

```redis
SET lock_key requestId NX EX 30
```

参数说明：

| 参数        | 作用              |
| --------- | --------------- |
| NX        | Key 不存在时才允许加锁   |
| EX        | 设置锁的过期时间        |
| requestId | 当前客户端唯一标识（UUID） |

由于加锁和设置过期时间在**同一条命令中完成**，整个过程具备原子性，因此不会出现锁永久存在的问题。

---

# 四、为什么 Value 必须唯一？

很多人解锁时会直接执行：

```redis
DEL lock_key
```

这样存在一个经典问题：

1. A 获取锁。
2. A 执行业务时间过长，锁自动过期。
3. B 获取到了新的锁。
4. A 执行 `DEL`，把 B 的锁删除了。

因此，**锁的 Value 必须保存当前客户端的唯一标识（如 UUID）**，用于判断锁的归属。

---

# 五、Lua 脚本保证安全解锁

正确的解锁流程应该是：

1. 判断锁是否属于自己；
2. 属于自己才删除。

对应 Lua 脚本如下：

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

为什么要使用 Lua？

因为 **判断（GET）+ 删除（DEL）** 必须是一个原子操作，否则在两条命令之间，锁可能已经被其他客户端重新获取。

---

# 六、锁过期怎么办？—— WatchDog 看门狗

锁设置了过期时间可以防止死锁，但也带来了新的问题。

例如：

- 锁超时时间：30 秒
- 业务执行时间：60 秒

那么锁会提前失效，其他客户端可能再次获取锁，导致多个线程同时操作共享资源。

Redisson 通过 **WatchDog（看门狗）机制** 解决了这个问题。

其原理是：

- 获取锁成功后启动后台续期任务；
- 默认每隔 **锁过期时间的 1/3** 自动续期一次；
- 当业务完成并主动释放锁后，停止续期。

这样，只要业务没有结束，锁就不会提前过期。

---

# 七、RedLock（红锁）

如果只有一台 Redis，当 Redis 宕机时，锁数据可能丢失。

RedLock 通过部署多个独立 Redis 节点来提高可靠性。

基本流程如下：

1. 向多个 Redis 节点依次执行加锁。
2. 在规定时间内，**超过半数节点加锁成功**，则认为获取锁成功。
3. 解锁时，向所有节点执行 Lua 解锁脚本。

RedLock 提高了容错能力，但实现和运维成本较高，因此普通业务场景使用并不多。

---

# 八、Redisson 为什么成为主流？

实际开发中，很少直接操作 Redis 命令，而是使用 **Redisson**。

Redisson 已经封装好了分布式锁的各种能力，包括：

- 原子加锁（`SET NX EX`）
- Lua 原子解锁
- WatchDog 自动续期
- 可重入锁
- 公平锁
- 读写锁
- RedLock

对于 Java 项目来说，它几乎是 Redis 分布式锁的事实标准。

---

# 九、可重入锁原理

所谓可重入锁，就是**同一个线程可以多次获取同一把锁**。

Redisson 内部会记录：

- 当前线程 ID
- 锁的重入次数

当同一个线程再次获取锁时，不会阻塞，而是将重入次数加一；每释放一次锁，重入次数减一，当次数减为 0 时，才真正删除锁。

---

# 十、总结

Redis 分布式锁的发展过程如下：

| 方案                         | 特点                  |
| -------------------------- | ------------------- |
| SETNX + EXPIRE             | 简单，但非原子，存在死锁风险      |
| SET NX EX                  | 官方推荐，原子加锁           |
| SET NX EX + Lua            | 防止误删锁，保证安全释放        |
| SET NX EX + Lua + WatchDog | 工业级方案，Redisson 默认实现 |
| RedLock                    | 多节点高可用方案，适用于特殊场景    |

> **一句话总结：**
> 
> Redis 分布式锁的核心思想是：**利用 `SET NX EX` 实现原子加锁，使用唯一 Value 标识锁归属，通过 Lua 脚本保证安全释放，再结合 WatchDog 自动续期解决锁提前过期问题。在 Java 项目中，通常直接使用 Redisson 即可。**
