### 1.说一下update set c =c +1 where id = 10这个sql的执行过程

#### 第一阶段：Server 层的流转（准备工作）

1. **连接器 (Connector)：** 你的客户端发起执行这条 SQL 的请求，连接器首先会校验你的用户权限，判断你有没有对这张表执行 Update 权限。

2. **分析器 (Analyzer)：** * **词法分析**：识别出 `update`、`set`、`where` 等关键字，以及表名、列名 `c` 和 `id`。
   
   - **语法分析**：检查这条 SQL 语句是否符合 MySQL 的语法规则。

3. **优化器 (Optimizer)：** 优化器会分析这条 SQL，决定如何执行效率最高。在这个例子中，因为条件是 `where id = 10`，优化器会决定**走 `id` 这个索引**（通常 `id` 是主键，也就是走聚簇索引直接定位数据）。

4. **执行器 (Executor)：** 优化器生成执行计划后，执行器开始真正执行。它会调用 InnoDB 存储引擎提供的 API 接口来获取数据和修改数据。

#### 第二阶段：InnoDB 引擎层的执行（核心考点）

这一部分是面试官最想听到的，特别是关于**日志**和**两阶段提交**的细节。

**1. 查询数据加载到内存 (Buffer Pool)**

- 执行器调用 InnoDB 引擎接口找 `id = 10` 的这一行。

- InnoDB 首先会去 **Buffer Pool（缓冲池）** 中查找这行数据所在的数据页：
  
  - **如果命中**：直接返回这行数据给执行器。
  
  - **如果未命中**：从磁盘的 `.ibd` 数据文件中将这一页读取出来，加载到 Buffer Pool 中，然后再返回给执行器。

**2. 更新内存数据 (Dirty Page)**

- 执行器拿到数据后，在自己的逻辑中把 `c` 的值加 1，然后把新数据再次调用 InnoDB 的接口写回去。

- 在修改数据之前，InnoDB 会先把这行数据**修改前的旧值**写入到 Undo Log 中。
  
  **目的**：如果事务执行失败或者你手动执行了 `ROLLBACK`，MySQL 就可以通过 Undo Log 恢复到修改前的状态。同时，Undo Log 也是实现 MVCC（多版本并发控制）的关键。

- InnoDB 收到新数据后，直接**在 Buffer Pool 内存中更新这行数据**。

- 注意：此时并没有立刻把数据写回磁盘！这个被修改过但还没写入磁盘的内存页被称为**脏页 (Dirty Page)**。MySQL 会在后台通过专门的线程择机将其刷入磁盘。

**3. 记录重做日志准备状态 (Redo Log - Prepare)**

- 虽然数据还没写盘，但为了防止数据库突然宕机导致内存中的脏页丢失，InnoDB 会利用 **WAL (Write-Ahead Logging) 技术**，先把这次修改操作的物理日志（例如“在某个数据页的某个偏移量处做了什么修改”）写入到 Redo Log Buffer，然后根据配置策略刷入磁盘中的 Redo Log 文件。

- 此时，Redo Log 进入 **`Prepare`（准备）状态**。

- 引擎告诉执行器：“我已经更新完了，你随时可以提交事务。”

**4. 记录归档日志 (Binlog)**

- 执行器收到引擎的回复后，生成这次操作的**逻辑日志 Binlog**（记录了 `update set c=c+1 where id=10` 这个操作或者修改后的具体行数据），并把 Binlog 写入磁盘。

**5. 提交事务 (Redo Log - Commit) —— 两阶段提交完成**

- Binlog 写入完成后，执行器调用引擎的“提交事务”接口。

- InnoDB 引擎将刚刚那个处于 `Prepare` 状态的 Redo Log 修改为 **`Commit` 状态**。

- 至此，这个 Update 语句才算真正执行完毕。



**为什么要用“两阶段提交”（Prepare & Commit）？** 这是为了保证 Redo Log（用于本机崩溃恢复）和 Binlog（用于主从复制和数据恢复）的**逻辑一致性**。如果 Redo Log 直接写 Commit，而 Binlog 没写完系统就崩溃了，重启后本机数据是最新的，但备库同步 Binlog 后数据还是旧的，就会产生数据不一致。两阶段提交配合崩溃恢复逻辑，可以完美解决这个问题。



**WAL 技术 (Write-Ahead Logging)** 先写日志（Redo Log），后写磁盘（数据文件）。写日志是顺序 I/O，速度极快；刷脏页是随机 I/O，较慢。这种设计极大地提升了数据库的并发写入性能。



### 2. Redo Log 刷入磁盘的策略有哪些？

Redo Log 并不是直接写到磁盘文件里的，而是先写到内存中的 Redo Log Buffer。至于什么时候把 Buffer 里的数据真正刷到磁盘（fsync），MySQL 提供了一个核心参数来控制：**`innodb_flush_log_at_trx_commit`**。

它有三种策略（也就是三个可选值）：

- **策略 `1`（默认值，最安全，性能最差）：** 每次事务提交时，都会把 Redo Log Buffer 里的日志立刻写入操作系统的 Page Cache，并且**强行调用 `fsync()` 刷入磁盘**。 *特点：* 真正意义上的 ACID 保证。即使 MySQL 进程崩溃或者服务器断电宕机，由于日志已经落盘，数据绝对不会丢。

- **策略 `0`（最不安全，性能最好）：** 每次事务提交时，什么都不做。由 InnoDB 的后台线程每隔 1 秒，把 Buffer 里的日志写入系统 Cache 并调用 `fsync()` 刷入磁盘。 *特点：* 如果 MySQL 进程崩溃或者服务器宕机，**最多会丢失 1 秒钟的事务数据**。

- **策略 `2`（折中方案）：** 每次事务提交时，只把日志写入操作系统的 Page Cache（不调用 `fsync` 落盘）。同样由后台线程每隔 1 秒去调用 `fsync()` 刷入磁盘。 *特点：* 如果只是 MySQL 进程崩溃，数据不会丢（因为日志已经交给了操作系统）；但如果服务器断电宕机，同样**最多会丢失 1 秒钟的事务数据**。
