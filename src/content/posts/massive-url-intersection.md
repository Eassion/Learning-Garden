## 问题

有两个文件，分别存放 50 亿个 URL。单个文件约 4TB，机器内存只有 4GB，如何找出两个文件中共同的 URL？

## 为什么不能直接放内存

50 亿个 URL 远远超过 4GB 内存。即使只存哈希值，也需要几十 GB 甚至更多空间，更不用说 URL 字符串本身。

所以这类题的关键不是“用什么集合 API”，而是：

> 把一个内存放不下的问题，拆成很多个内存放得下的小问题。

## 推荐方案：哈希分桶

核心思路：

1. 对两个文件中的每个 URL 计算同一个哈希值。
2. 按 `hash(url) % N` 把 URL 分到 N 个小文件中。
3. 两个原始文件都使用完全相同的分桶规则。
4. 这样相同的 URL 一定会落到编号相同的桶里。
5. 之后只需要逐桶比较，例如比较 `A_0` 和 `B_0`、`A_1` 和 `B_1`。

如果每个原始文件是 4TB，内存只有 4GB，桶数量可以设置得足够大，比如 4096 或 8192。这样每个桶平均大小大约是 1GB 或更小，单桶就有机会放入内存处理。

## 具体步骤

第一步，分桶：

```text
fileA:
  url -> hash(url) % N -> A_i

fileB:
  url -> hash(url) % N -> B_i
```

第二步，逐桶求交集：

```text
for i in 0..N-1:
  把 A_i 中的 URL 放入 HashSet
  顺序扫描 B_i
  如果 URL 在 HashSet 中，就输出到 result
```

因为相同 URL 的哈希值相同，所以它们一定会进入相同编号的桶。不同编号的桶之间不可能存在相同 URL，因此不需要跨桶比较。

## 内存仍然不够怎么办

如果某些桶因为数据倾斜仍然过大，可以继续处理：

- 增大桶数量 N。
- 对过大的桶做二次哈希分桶。
- 先对桶内 URL 去重，减少重复数据。
- 使用外部排序代替 HashSet。

实际工程里，桶大小不能只按平均值估算，还要考虑热点 URL、哈希分布和文件系统小文件数量。

## 另一种方案：外部排序

也可以分别对两个大文件做外部排序：

1. 每次读取一部分 URL 到内存。
2. 内存中排序后写成有序小文件。
3. 对小文件做多路归并，得到两个整体有序的大文件。
4. 用双指针顺序扫描两个有序文件，遇到相同 URL 就输出。

外部排序的优点是顺序读写友好，结果天然有序。缺点是排序成本较高，通常会比哈希分桶更慢一些。

## 两种方案对比

| 方案 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- |
| 哈希分桶 | 思路直接，求交集快，容易并行 | 需要处理数据倾斜和桶文件管理 | 只关心共同 URL |
| 外部排序 | 顺序 IO 友好，结果有序 | 排序成本高，实现链路更长 | 需要有序结果或后续范围扫描 |

## 面试回答模板

这道题不能直接把 URL 放进内存，因为两个文件都是 TB 级别。我的做法是用哈希分桶。

先对两个文件里的 URL 使用同一个哈希函数，按照 `hash(url) % N` 分成很多小文件。相同 URL 一定会进入同一个编号的桶。然后逐个比较同编号的桶：把 A 桶加载到 HashSet，扫描 B 桶，如果存在就输出。只要 N 设得足够大，每个桶就能控制在 4GB 内存以内。

如果某些桶仍然太大，说明有数据倾斜，可以增加桶数量，或者对大桶继续二次分桶。另一种方案是外部排序，把两个文件分别排序后用双指针扫描求交集，但如果只是找共同 URL，哈希分桶通常更直接。

## 复杂度

哈希分桶方案整体需要顺序扫描原始文件，再逐桶扫描一次，时间复杂度接近 `O(n + m)`。额外空间主要是磁盘上的分桶文件，内存只需要容纳一个桶和少量缓冲区。

这类海量数据题的核心套路是：外存拆分、局部处理、最后合并结果。
