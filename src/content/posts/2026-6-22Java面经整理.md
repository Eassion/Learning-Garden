### 一、JVM运行时数据区

#### a、介绍一下 JVM 的运行时数据区（内存结构）？哪些是线程共享的，哪些是私有的？

JVM 在运行 Java 程序时，会把内存划分为 5 个区域。

为了逻辑清晰，我习惯把它们按照生命周期和是否线程共享**分为两类：**

**第一类是**线程私有的，随线程生灭，**不需要垃圾回收**（GC）； 第二类是**线程共享的**，也是 GC 重点关注的区域。

下面我具体展开说一下。

首先是**线程私有**的区域，包含三个部分：

1. **程序计数器**： 它的作用是记录当前线程执行到哪一行字节码了。
- *为什么需要它？* 因为 CPU 会在不同线程间切换，当切回来的时候，线程得知道自己刚才执行到哪儿了。

- *特点*：这是 JVM 中唯一一个不会发生内存溢出（OOM）的区域。
2. **虚拟机栈**： 这是我们平时最常说的栈，它是 Java 方法执行的内存模型。
- 每个方法在执行时都会创建一个**栈帧**，里面存了局部变量表、操作数栈等信息。方法调用入栈，执行完出栈。

- *异常*：如果递归太深，会爆出 `StackOverflowError`；如果申请不到内存，会爆出 `OOM`。
3. **本地方法栈（Native Method Stack）**： 这个和虚拟机栈逻辑一样，只不过它是服务于 `Native` 方法（比如 C++ 写的底层方法）的。
- *注*：在 HotSpot 虚拟机中，其实直接把这两个栈合二为一了。”

接下来是**线程共享**的区域，这也是出现问题最多的地方

1. **堆**： 这是 JVM 内存中**最大**的一块。**几乎所有的对象实例和数组**都在这里分配内存。
- *关注点*：它是垃圾回收器（GC）的主要工作场所。我们常说的‘新生代’、‘老年代’划分，就是指的这里。

- *异常*：如果对象太多塞满了，就会报 `OutOfMemoryError: Java heap space`。
2. **方法区**： 它存储了已被虚拟机加载的**类信息、常量、静态变量**等数据。
- *版本差异（亮点）*： 在 **JDK 1.7 及以前**，它在实现上叫‘永久代’（PermGen），且位于堆内存中，容易导致 OOM。 而在 **JDK 1.8 及以后**，JVM 移除了永久代，改用在本地内存（Native Memory）中实现的 **元空间（Metaspace）**。这样做的好处是方法区不再受堆内存大小的限制，减少了 OOM 的风险。”

总结一下：**栈管运行，堆管存储。** 栈是线程私有的，堆和方法区是线程共享的。

#### b、这些区域里，哪些会发生 OOM（内存溢出）

除了**程序计数器**以外，其他 4 个区域都有可能发生 OOM。

1. **堆**：最常见。对象太多且无法回收时，报 `OutOfMemoryError: Java heap space`

2. **方法区**：加载的类太多（比如 Spring 动态代理生成太多类），报 `OutOfMemoryError: Metaspace`。

3. **栈**：虽然平时更多报的是 `StackOverflowError`（递归太深），但如果线程请求的栈深度大于允许深度，或者创建太多线程导致内存耗尽，也会报 OOM。

#### c、**简单说一下JVM里栈和堆的区别是什么？**

可以从三个维度来区分：

1. **存储内容**：
- **栈**主要存储**局部变量**（基本数据类型、对象的引用）和方法调用的上下文。

- **堆**存储的是**实际的对象实例**（`new` 出来的东西）。
2. **内存管理**：
- **栈**是自动管理的，方法结束了自动出栈释放，**不需要 GC**。

- **堆**是手动管理（C++）或 GC 管理（Java）的，是垃圾回收的主要场所。
3. **物理地址**：
- **栈**通常是连续的内存空间（速度快）。

- **堆**是不连续的内存空间，通过链表等方式维护（灵活但稍慢）。

#### d、**JDK 1.8 在内存区域上有什么重大变化？为什么要用元空间替代永久代**

最大的变化是：**移除了永久代，引入了元空间。** 在 JDK 1.7 及之前，方法区是堆内存的一部分（永久代）。 而在 JDK 1.8 中，元空间使用的是**本地内存**，不再占用 JVM 的堆内存。

**为什么要替换？主要有两个原因**：

1. **避免 OOM**：永久代的大小是启动时固定的，很难调整。现在的应用（尤其是 Spring 这种大量使用动态代理的框架）会动态生成很多类，很容易把永久代撑爆。而元空间直接使用服务器的物理内存，上限大得多。

2. **合并 HotSpot 和 JRockit**：当年 Oracle 收购了 JRockit，JRockit 里面就没有永久代，为了代码统一，就把 HotSpot 的永久代也去掉了。

**元空间既然用的是本地内存，是不是就永远不会 OOM 了**

**不是的**，依然会 OOM。 虽然本地内存很大，但不是无限的。 如果在生产环境中，我们没有限制元空间大小（默认是无限），且程序有 Bug 导致动态疯狂生成类，可能会把服务器所有的物理内存吃光，导致整个服务器宕机（被系统 Kill 掉）。 所以线上环境我们通常还是会配置 `-XX:MaxMetaspaceSize` 来兜底。

### 二、垃圾回收算法

#### a、**JVM 如何判断一个对象是否可以被回收？**

现在主流 JVM 采用的是 **可达性分析算法**，而不是单纯的引用计数。

它的思路是：从一组称为 **GC Roots** 的根对象出发，沿着引用关系向下搜索。  
如果一个对象从 GC Roots 出发不可达，那么这个对象就可以被判定为垃圾对象。

常见的 GC Roots 包括：

- 虚拟机栈中引用的对象

- 方法区中类静态属性引用的对象

- 方法区中常量引用的对象

- 本地方法栈 JNI 引用的对象

之所以不用引用计数，核心原因是 **它无法解决循环引用问题**。

#### b、常见的垃圾回收算法

主要有四种：

**第一种，标记-清除。**  

先标记存活对象，再清理未标记对象。优点是实现简单；缺点是会产生内存碎片。

**第二种，复制算法。**  

把内存分成两块，每次只使用一块；回收时把存活对象复制到另一块。优点是没有内存碎片；缺点是浪费一半空间。

**第三种，标记-整理。**  

先标记存活对象，再把存活对象向一端移动，最后直接清理边界外内存。优点是没有碎片；缺点是整理对象成本较高。

**第四种，分代收集。**  

这不是一种具体算法，而是一种组合思想：年轻代和老年代的对象特点不同，所以采用不同算法更合适。

#### c、**为什么 JVM 要做分代回收？**

因为不同区域的对象生命周期分布非常不一样。

年轻代的对象特点是：**创建多、死亡快、存活率低**。  

所以年轻代更适合复制算法，回收效率高。

老年代的对象特点是：**存活率高、对象体积更大**。  

如果还用复制算法，成本就太高了，所以老年代通常更适合标记-清除或标记-整理。

所以分代回收的本质是：  

**针对不同对象生命周期，采用更匹配的回收策略。**

#### d、**什么是三色标记法？**

三色标记法是并发标记过程中常用的一种抽象模型，用来描述对象的扫描状态。

- **白色**：还没被标记，理论上可能是垃圾

- **灰色**：已经被标记，但它引用的对象还没扫描完

- **黑色**：已经被标记，而且它引用的对象也都扫描完成

并发标记时，GC 线程和用户线程可能同时修改对象引用关系。  
如果处理不好，就可能出现“本来应该存活的对象被漏标”的问题。

因此现代垃圾回收器通常会配合：

- **增量更新**

- **SATB（原始快照）**

来修正并发标记带来的问题。

#### e、**常见的垃圾回收器有哪些？怎么选？**

常见垃圾回收器可以按代际来看：

**年轻代** 常见有：

- Serial

- ParNew

- Parallel Scavenge

**老年代** 常见有：

- Serial Old

- CMS

- Parallel Old

**整堆/区域化回收器** 常见有：

- G1

- ZGC

- Shenandoah

如果是 JDK8 时代的常见答案，面试最常问的是 **CMS 和 G1**；  

如果是更新一点的版本，也可能会问 **ZGC**。

| 垃圾收集器                                | 特点                                                                                                                                                          | 适用场景                                                                                                                                    |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| **Serial GC**                        | 单线程收集，简单，GC 时会 Stop The World                                                                                                                               | 小内存、单核、客户端程序                                                                                                                            |
| **ParNew GC**                        | Serial 的多线程版本，主要用于新生代                                                                                                                                       | 以前常和 CMS 搭配                                                                                                                             |
| **Parallel Scavenge / Parallel Old** | 多线程，追求吞吐量，也叫吞吐量优先收集器                                                                                                                                        | 后台计算、批处理任务                                                                                                                              |
| **CMS GC**                           | 并发标记清除，目标是降低停顿时间                                                                                                                                            | 老版本常用，JDK 14 已移除 ([OpenJDK](https://openjdk.org/jeps/363?utm_source=chatgpt.com "JEP 363: Remove the Concurrent Mark Sweep (CMS) ...")) |
| **G1 GC**                            | 分 Region 管理堆，兼顾吞吐和低停顿，是很多 JDK 配置下的默认选择 ([Oracle Docs](https://docs.oracle.com/en/java/javase/21/gctuning/available-collectors.html "Available Collectors")) | 服务端应用、一般后端系统                                                                                                                            |
| **ZGC**                              | 超低延迟 GC，停顿时间通常很短，适合大堆低延迟场景；JDK 21 引入分代 ZGC ([OpenJDK](https://openjdk.org/jeps/439?utm_source=chatgpt.com "JEP 439: Generational ZGC"))                     | 大内存、低延迟系统                                                                                                                               |
| **Shenandoah GC**                    | 低停顿收集器，很多 GC 工作与用户线程并发执行 ([OpenJDK](https://openjdk.org/jeps/379?utm_source=chatgpt.com "JEP 379: Shenandoah: A Low-Pause-Time Garbage ..."))               | 低延迟服务                                                                                                                                   |
| **Epsilon GC**                       | 只分配内存，不真正回收                                                                                                                                                 | 性能测试、极短生命周期程序                                                                                                                           |

#### f、**G1 的特点是什么？**

G1 的核心特点是把整个堆划分成很多 Region，不再机械区分连续的年轻代和老年代空间。  

它会优先回收“垃圾最多、收益最高”的 Region，因此叫 Garbage First。

它的优势是：

- 更适合大堆

- 可预测停顿时间

- 整体吞吐和停顿之间比较均衡

### 三、类加载

#### a、讲一下类加载的整个流程？

Java 的类加载机制是指把 `.class` 文件加载到内存，并进行校验、解析和初始化，最终形成可以被虚拟机直接使用的 Java 类型。 整个过程分为 5 个阶段：

1. 加载 ：通过类的全限定名（包名+类名）找到 `.class` 二进制流，把它读进内存，并在堆中生成一个 `java.lang.Class` 对象作为访问入口。

2. 验证 ：检查这个字节码是不是合法的（比如文件头是不是 `0xCAFEBABE`），防止恶意代码破坏 JVM。

3. **准备 【重点】**：为类的**静态变量（static）分配内存，并设置默认初始值**。
- *注意*：这里是设置默认值（比如 `int` 为 0，`Object` 为 null），而不是代码里写的那个值。
4. **解析**：把常量池里的**符号引用**（比如仅仅是个字符串 "ArrayList"）替换为**直接引用**（内存中的实际地址指针）。

5. **初始化 【重点】**：这是类加载的最后一步。
   
   这时候才真正执行类中定义的 Java 代码（也就是 `static` 代码块和 `static` 变量的赋值操作）。

#### b、什么时候会触发类的初始化？

JVM 规定了只有 **主动引用** 时才会初始化，常见的场景有：

1. `new` 一个对象时。

2. 读取或设置一个类的静态字段（static）时。

3. 调用类的静态方法时。

4. 对类进行反射调用（`Class.forName`）时。

5. 初始化一个子类时，如果父类还没初始化，会先触发父类的初始化。

### 二、双亲委派模型

#### a、什么是类加载器？

首先，类加载器的作用就是把编译好的 `.class` 字节码文件加载到 JVM 内存中。 在 JDK 8 及之前的版本中，主要有三个核心的类加载器，它们呈层级关系：

1. **启动类加载器**： 它是最顶层的。负责加载 JDK 的**核心类库**（即 `JAVA_HOME/lib` 目录下的类，比如 `String`、`System` 等）。
- *注意点*：它是由 C++ 实现的，所以我们在 Java 代码里拿不到它的引用（会返回 null）。
2. **扩展类加载器** ： 它负责加载 JDK 的**扩展库**（`JAVA_HOME/lib/ext` 目录）。

3. **应用类加载器**： 这是我们平时接触最多的。它负责加载 `Classpath` 下的类，也就是**我们自己写的代码**和引入的第三方 jar 包。”

父加载器是子加载器的父类吗？

**不是继承关系**，而是**组合关系**。 它们在代码层面并没有 `extends` 关系，而是通过一个 `parent` 成员变量持有父加载器的引用。这是一种包含关系，不是类继承

#### b、什么是双亲委派模型？为什么要这么设计？

所谓双亲委派模型，其实不是指有两个父母，而是指‘父辈委托’机制。 它的核心逻辑可以概括为八个字：‘**向上委托，向下加载**’。

具体流程是这样的： 当一个加载器（比如应用类加载器）收到类加载请求时，它**不会自己先去加载**，而是把这个请求**原封不动地委托给父加载器**去尝试。 每一层都是如此，直到请求传送到最顶层的 **Bootstrap ClassLoader**。

- 如果父加载器在自己的搜索范围内找到了这个类，就成功返回；

- 只有当父加载器反馈‘我加载不了’（找不到）时，子加载器才会尝试自己去加载。

这么设计主要有两个目的：**安全**和**避免重复加载**。最核心的是**安全**。

举个最经典的例子： 假设有人恶意写了一个 `java.lang.String` 类，里面植入了破坏系统的代码。

- **如果没有双亲委派**：应用类加载器可能直接就加载了这个恶意的 String 类，那整个系统的 String 就被替换了，JVM 也就崩溃了。

- **有了双亲委派**：当加载请求向上委托到 Bootstrap ClassLoader 时，它发现自己负责的 `rt.jar` 里已经有了 `java.lang.String`，它就会直接加载 JDK 自带的那个安全的 String 类。

这样，恶意的类就永远没有机会被加载进来。这就保证了 **Java 核心类库的安全性（沙箱安全机制）**

#### c、**有什么场景需要打破双亲委派，怎么打破**

双亲委派不是强制约束，而是一种建议  

场景一：Tomcat  

●Tomcat 上可能部署了两个 Web 应用。应用 A 依赖 Spring 4，应用 B 依赖 Spring 5。●如果遵循双亲委派，类加载器只会加载一份 Spring（假设是 4）。那应用 B 也是用的 Spring 4，启动就报错了。  

●解决：Tomcat 自定义了 WebAppClassLoader，它重写了 loadClass 方法。优先自己加载 WEB-INF/classes 下的类，加载不到再给父类。这样就实现了应用间的隔离。  
场景二：JDBC  

●JDBC 的接口（java.sql.Driver）是 Java 核心包，由 Bootstrap 加载。  

●但具体的驱动实现（如 MySQL Driver）是厂商提供的第三方 Jar 包，在 Classpath 下，由 Application 加载器加载。  

●矛盾：Bootstrap 加载的代码（DriverManager）需要去调用 Application 加载的代码（MySQL Driver），这违反了双亲委派（父调子）。  

●解决：利用 线程上下文类加载器 (Thread Context ClassLoader)。核心类通过 Thread.currentThread().getContextClassLoader() 拿到应用类加载器，去加载第三方的实现类。
