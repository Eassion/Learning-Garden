### 一、集合

**1.应该是先大而光地罗列一下**

Java集合框架主要分为两条主线，Collection接口和Map接口，Collection下主要有三个字接口：List（有序、可重复）、Set（无序、唯一）和Queue（队列），Map接口则是存储键值对的结构。

在具体实现上，常用的有ArrayList、LinkedList、HashSet、TreeSet，以及最核心的HashMap、ConcurrentHashMap和TreeMap

**2.然后有侧重地介绍自己比较熟悉的**

比如我在项目里实现哪块为了什么什么目的（某个数据结构的优势），使用了什么集合，是怎么使用的。

**3.在这之后就会顺着你的使用展开聊了**



#### A、ArrayList与LinkedList的区别

在底层结构上，ArrayList是基于动态数组的，在内存中是连续存储的；而LinkedList是基于双向链表的（JDK1.7之后），它的节点在内存中是分散存储的，通过指针连接。

**性能差异：**

**查询**：ArrayList支持随机访问，通过下标能直接算出地址，复杂度是O(1)。Linked需要从头遍历，复杂度是O(n)。

**增删**：LinkedList在已知节点的情况下，增删只需改动指针，是O(1)；ArrayList如果在中间增删，需要挪动后续所有元素，是O(n)。

#### B、你在项目中怎么选这两种数据结构

绝大多数场景（90%）我都会选ArrayList

除了支持快速查询，ArrayList的连续内存对CPU缓存（Cache Line）非常友好，遍历速度极快；而LinkedList内存分散，容易造成缓存未命中

只有在“频繁地在头部或中间插入删除，且几乎不用随机访问”这种极少数特殊场景下，我才会考虑LinkedList。

#### C、在List中间插入1000万条数据，谁更快？

这要看情况。如果是用for循环根据下标i去插入，ArrayList反而更快，因为LinkedList每次都要get(i)遍历找到位置，这一步就是O(n)，总复杂度变成O(n²)。只有用Iterator遍历并插入，才能发挥O(1)的插入优势。

#### D、ArrayList 是线程安全的吗？如果不安全怎么办 ？

不是线程安全的。多线程下建议使用`CopyOnWriteArrayList`（适合读多写少）或者用 `Collections.synchronizedList` 包装一下

### E、ArrayList的扩容机制

ArrayList 的扩容主要发生在 `add` 操作时。

在 JDK 1.8 中，如果使用无参构造器，它初始化的是一个**空数组**（懒加载）。只有当第一次 `add` 添加元素时，才会初始化为默认容量 **10**。

当当前元素个数等于数组容量时，就会触发扩容。

新容量通常是旧容量的 **1.5 倍**。*源码细节*：`int newCapacity = oldCapacity + (oldCapacity >> 1);` （右移一位相当于除以 2）。

计算好新容量后，底层会调用 `Arrays.copyOf`（本质是 `System.arraycopy`），在堆中创建一个新数组，把老数据复制过去，然后丢弃老数组。

**如何优化？**

最直接的优化就是**预估容量**。 如果我能预知列表大概要存 1000 个对象，我就直接 `new ArrayList<>(1000)`。 这样就避免了中间多次的扩容（10 -> 15 -> 22...），既省了时间也减少了数组碎片的产生。

#### F、HashMap的底层原理

HashMap 的底层结构在JDK 1.7 和 1.8 中是有区别的，我分开来讲，JDK 1.7采用的是**数组+链表**的结构，而 JDK 1.8 做了优化，采用了**数组+链表+红黑树**的结构。

**具体的插入流程（put）是这样的**

1. 首先，根据 Key 计算 Hash 值，经过扰动函数处理后，通过(数组长度-1) & 键的哈希值 公式确定它在数组中的下标。

2. 如果该位置为空，直接插入。

3. 如果该位置已经有数据了（发生Hash冲突），会遍历链表检查Key是否存在：
- 如果Key相同，就覆盖Value。

- 如果Key不同，JDK 1.8 会判断当前节点是链表还是红黑树。如果是链表，采用**尾插法**追加到末尾；如果是红黑树，则调用树的插入方法。
4. **重点是**，在 JDK 1.8 中，当链表长度超过 8 且数组长度大于等于 64 时，链表会转换为红黑树，这样能将查找复杂度从 O(n) 降低到 O(log n)，解决链表过长导致的性能下降问题。

#### G、为什么 JDK 1.8 要用尾插法？1.7 的头插法有什么问题

JDK 1.7 用头插法是因为作者认为最新插入的数据被访问的概率更大。 但在**多线程并发扩容**时，头插法会改变链表的顺序，容易导致**环形链表**，从而引发死循环（CPU 100%）。 1.8 改用尾插法，扩容后链表顺序保持不变，就解决了这个问题。

#### H、**HashMap的扩容机制是怎么样的**

HashMap 默认初始容量是 **16**，加载因子是 **0.75**。 当元素数量超过 `容量 * 0.75` 时，就会触发扩容，容量变为原来的**2倍**。 扩容时需要重新计算每个元素在数组中的位置。不过 JDK 1.8 做了优化，不需要重新 Hash，只需要看 hash 值新增的那一位比特是 0 还是 1。如果是 0，位置不变；如果是 1，位置变为 `原索引 + 原容量`。这样效率更高。

#### I、为什么 HashMap 的容量必须是 2 的幂

为了支持两个优化：

1. **极速寻址**：只有当容量是 2 的幂时，`hash % 容量` 才能等价于位运算 `hash & (容量 - 1)`，位运算比取模快得多。

2. **扩容优化**：就是刚才讲的，利用高位比特判断位置，避免重新 Hash。

#### J、HashMap 是线程安全的吗

HashMap **不是**线程安全的。在多线程环境下使用它，主要会产生两个严重问题：

1. **数据丢失（覆盖）**：这是最常见的问题。
- 在 JDK 1.7 和 1.8 中都存在。当两个线程同时进行 `put` 操作，并且 Key 的 Hash 值冲突时，它们可能会同时判断“当前位置为空”。

- 于是，线程 A 刚准备插入，线程 B 抢先插入了。等线程 A 恢复执行时，它不知道该位置已经被占了，就会直接写入，把线程 B 的数据**覆盖**掉。
2. **死循环（CPU 100%）**：这是 JDK 1.7 的经典 Bug。
- JDK 1.7 在扩容时采用的是**头插法**。

- 在并发扩容时，如果链表结构被破坏，可能会导致链表成环（A指向B，B又指向A）。

- 当下次 `get` 操作遍历这个链表时，程序就会进入死循环，导致服务器 CPU 飙升。

- *补充*：虽然 JDK 1.8 改用了**尾插法**解决了死循环问题，但数据覆盖的问题依然存在，所以它依然是不安全的。

补充，生产环境要使用线程安全的map有三种方案

1. **Hashtable**：虽然安全，但它给所有方法都加了 `synchronized`，锁粒度太粗，全表只有一把锁，效率极低，现在基本没人用了。

2. **Collections.synchronizedMap**：它是对 HashMap 做了一层包装，本质上也是用 synchronized 锁住整个 Map，并发性能依然不高。

3. **ConcurrentHashMap**：这是**最佳选择**。它通过降低锁粒度（分段锁或节点锁）实现了高效的并发。

#### K、ConcurrentHashMap的实现原理

ConcurrentHashMap 的核心思想是**分而治之**，尽量缩小锁的范围。JDK 1.7 和 1.8 的实现差异很大：

**1. JDK 1.7：分段锁**

- **结构**：它由一个 `Segment` 数组组成。每个 `Segment` 本质上继承了 `ReentrantLock`。

- **原理**：它把一个大的 HashMap 切分成了 16 个小段（Segment）。

- **并发度**：理论上支持 16 个线程并发写入，只要它们访问的是不同的段，就不会互斥。

- *缺点*：并发度受限于 Segment 的个数，且 Segment 对象本身占用不少内存。

**2. JDK 1.8：CAS + Synchronized**

- **结构**：摒弃了 Segment，结构和 HashMap 1.8 一样（数组+链表+红黑树）。

- **原理**：

- 它**直接锁链表/红黑树的头节点**。

- **如果是第一次插入**（桶为空）：利用 **CAS** (Compare And Swap) 指令进行无锁插入，速度极快。

- **如果发生冲突**（桶不为空）：使用 **synchronized** 锁住当前桶的头节点。

- **优势**：锁粒度细化到了每一个 Node（桶）。只要 Hash 不冲突，几千个线程都能并发写，性能比 1.7 提升巨大。

### L、为什么 JDK 1.8 要把 ReentrantLock 换成 synchronized？synchronized 不是性能很差吗

这其实是一个误区。

1. **JDK 优化**：从 JDK 1.6 开始，JVM 对 synchronized 做了大量优化（偏向锁、轻量级锁、锁消除等），在低竞争下，它的性能已经不输 ReentrantLock 了。

2. **节省内存**：ReentrantLock 需要创建额外的 Segment 对象，而 synchronized 直接锁住 Node 对象头，不需要额外内存。

3. **自动释放**：synchronized 自动释放锁，而 Lock 必须手动 unlock，代码更简洁安全。

### M、ConcurrentHashMap 的 Key 和 Value 能为 null 吗？为什么？

**坚决不能**（HashMap 可以）。

**原因**：主要是为了避免**二义性**（Ambiguity）。 假设我 `map.get(key)` 返回了 null。

- 在单线程（HashMap）中，我可以再调用 `map.containsKey(key)` 来确认是‘没这个键’还是‘值为 null’。

- 但在多线程（ConcurrentHashMap）中，这两次调用之间，可能有另一个线程把数据改了。所以我没法判断最初那个 null 到底是什么意思。 为了系统健壮性，并发集合直接禁止了 null。
